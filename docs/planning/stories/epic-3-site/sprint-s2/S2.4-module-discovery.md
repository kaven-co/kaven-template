# S2.4 — Module Discovery

> **Epic:** 3 (Site — Customer Portal) | **Sprint:** S2 | **Agent:** @dev
> **Priority:** P1
> **Depends on:** M1.6

---

## Contexto

O marketplace de modulos e um diferencial chave do Kaven — permite que desenvolvedores expandam o framework com modulos oficiais e comunitarios. A pagina de module discovery no kaven-site e a vitrine desse marketplace: permite browsear, filtrar, buscar, e ver detalhes de cada modulo antes de instalar via CLI.

A pagina consome dados do marketplace API (`GET /modules`) e nao faz instalacao diretamente — o CTA e copiar o comando CLI para o clipboard. Para modulos comunitarios pagos, o CTA pode levar a um checkout. Para modulos oficiais incluidos no tier, o CTA e simplesmente o comando de instalacao.

---

## User Story

**Como** desenvolvedor usando o Kaven,
**Quero** browsear, buscar, e ver detalhes de modulos disponiveis no marketplace,
**Para que** eu encontre e instale modulos que adicionam funcionalidade ao meu projeto sem desenvolvimento custom.

## Acceptance Criteria

### Module Listing Page
- [ ] Route `/modules` renders the module discovery page (public — no auth required to browse)
- [ ] Grid of module cards showing all available modules
- [ ] Search bar at top with real-time filtering (debounced, 300ms)
- [ ] Search matches: module name, description, and tags
- [ ] Category filter sidebar (desktop) or dropdown (mobile):
  - All, Authentication, Payments, Email, Analytics, Design System, Storage, Notifications, AI, Community
- [ ] Sort options: "Most Popular", "Recently Updated", "Alphabetical"
- [ ] Results count shown: "Showing 12 of 45 modules"
- [ ] Pagination or infinite scroll if more than 20 modules per page

### Module Card
- [ ] Each module card displays:
  - Module icon/logo (placeholder if none)
  - Module name (e.g., "kaven-payments", "kaven-auth-magic-link")
  - Short description (max 2 lines, truncated with ellipsis)
  - Version badge: "v1.2.3"
  - Tier badge: which tiers include this module (Starter/Complete/Pro) — or "Community" for third-party
  - Install count: "1.2k installs"
  - Author: "Kaven" (official) or author name (community)
  - Last updated: relative time ("3 days ago")
- [ ] Card hover: subtle elevation/shadow effect
- [ ] Card click: navigates to module detail page

### Module Detail Page
- [ ] Route `/modules/:slug` renders the module detail page
- [ ] Module detail header:
  - Icon, name, version, author, tier badges
  - Install count, last updated, license type
  - "Official" or "Community" badge
- [ ] Full description section (Markdown rendered — supports headings, lists, code blocks, images)
- [ ] Installation section with CLI command: `kaven module add {slug}`
  - Copy button for CLI command
  - Prerequisites listed (e.g., "Requires kaven-cli v1.0+")
- [ ] Versions tab: list of versions with release notes and dates
  - Expandable release notes per version
  - "Latest" badge on most recent
- [ ] Dependencies section: list of other modules this module depends on (with links)
- [ ] Screenshots/previews section: image gallery if the module has UI components
- [ ] Configuration section: environment variables and config options needed
- [ ] Related modules: "You might also like" based on category

### Community Modules
- [ ] Community modules have distinct visual treatment (different card border color or badge)
- [ ] Community module cards show: author name, avatar, "Community" badge
- [ ] If community module is paid: show price and "Purchase" CTA
- [ ] "Purchase" CTA → Paddle checkout for the specific module
- [ ] If community module is free: same "Install" CTA as official modules

### Data & API
- [ ] Module list fetched from marketplace API: `GET /modules?category=&search=&sort=&page=`
- [ ] Module detail fetched from: `GET /modules/:slug`
- [ ] Module versions fetched from: `GET /modules/:slug/versions`
- [ ] Loading skeletons for both list and detail pages
- [ ] Error handling with retry button
- [ ] Cache module list client-side (SWR) to avoid refetching on back navigation

## Technical Notes

- **Data types**:
  ```typescript
  // lib/api/modules.ts
  interface Module {
    id: string;
    slug: string;
    name: string;
    description: string;
    fullDescription: string;  // Markdown
    version: string;
    author: {
      name: string;
      avatarUrl: string;
      isOfficial: boolean;
    };
    category: string;
    tags: string[];
    tiers: ('starter' | 'complete' | 'pro')[];
    installCount: number;
    lastUpdated: string;
    iconUrl: string | null;
    screenshots: string[];
    dependencies: string[];  // module slugs
    configOptions: ConfigOption[];
    price: number | null;     // null = free/included
    currency: string | null;
  }
  interface ModuleVersion {
    version: string;
    releaseNotes: string;  // Markdown
    date: string;
    isLatest: boolean;
  }
  ```
- **Search implementation**: Client-side filtering for initial MVP (if module count < 100). Server-side search via API query param for scalability.
- **Markdown rendering**: Use `react-markdown` with `remark-gfm` for module descriptions and release notes. Syntax highlighting with `rehype-highlight` or `rehype-shiki`.
- **URL structure**: `/modules` for listing, `/modules/[slug]` for detail. Use Next.js dynamic routes.
- **File structure**:
  ```
  app/modules/
  ├── page.tsx                          # Module listing page
  ├── loading.tsx                       # Loading skeleton
  ├── [slug]/
  │   ├── page.tsx                      # Module detail page
  │   └── loading.tsx                   # Detail loading skeleton
  components/modules/
  ├── ModuleGrid.tsx                    # Grid layout for module cards
  ├── ModuleCard.tsx                    # Individual module card
  ├── ModuleSearch.tsx                  # Search bar with debounce
  ├── CategoryFilter.tsx               # Category sidebar/dropdown
  ├── SortSelect.tsx                    # Sort dropdown
  ├── ModuleDetail.tsx                  # Full detail view
  ├── ModuleVersions.tsx               # Version history list
  ├── ModuleScreenshots.tsx            # Screenshot gallery
  ├── InstallCommand.tsx               # CLI command with copy
  └── RelatedModules.tsx               # Related module suggestions
  lib/api/
  └── modules.ts                        # Modules API client
  ```
- **SEO**: Module detail pages should have dynamic `generateMetadata` for unique title/description per module. OG images can be auto-generated with module name.
- **Performance**: Module list page uses pagination or virtual scrolling for large lists. Images lazy-loaded.
