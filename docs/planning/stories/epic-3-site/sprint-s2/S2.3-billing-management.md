# S2.3 — Billing Management

> **Epic:** 3 (Site — Customer Portal) | **Sprint:** S2 | **Agent:** @dev
> **Priority:** P1
> **Depends on:** M2.5

---

## Contexto

Clientes precisam gerenciar seus pagamentos, ver historico de faturas, fazer upgrade/downgrade de tier, e cancelar assinatura — tudo sem precisar enviar email para suporte. O Paddle e o gateway de pagamentos para o marketplace, e ele oferece uma Customer Portal embeddable que lida com metodo de pagamento, mas o Kaven precisa de uma UI propria para exibir detalhes da assinatura, acoes de tier change, e historico de invoices de forma integrada com o resto do portal.

A dependencia em M2.5 (marketplace billing endpoints) significa que os dados vem do marketplace API, que por sua vez consulta a Paddle API. O kaven-site nunca fala diretamente com a Paddle API.

---

## User Story

**Como** cliente com assinatura ativa,
**Quero** ver meus detalhes de pagamento, faturas, e poder mudar ou cancelar meu plano,
**Para que** eu tenha controle total sobre minha assinatura sem depender de suporte.

## Acceptance Criteria

### Subscription Overview
- [ ] Route `/billing` renders the billing management page (protected route)
- [ ] Current subscription card displaying:
  - Current tier name and badge (Starter/Complete/Pro)
  - Billing cycle: Monthly or Yearly
  - Price per cycle: "$99/month" or "$990/year"
  - Next billing date: formatted date (e.g., "March 15, 2026")
  - Subscription status: Active, Past Due, Cancelled (with end date)
  - Subscription start date
- [ ] If no active subscription: show "No active subscription" with CTA to pricing page

### Upgrade/Downgrade
- [ ] "Change Plan" button on subscription card
- [ ] Click opens plan comparison modal/sheet showing:
  - Current plan highlighted
  - Other available plans with price differences (e.g., "+$180/mo for Pro")
  - Feature differences between current and target plan
  - Proration note: "You'll be charged the prorated difference immediately"
- [ ] "Upgrade to {Plan}" button in modal → calls marketplace API `POST /billing/change-plan`
- [ ] API creates Paddle checkout session for plan change → opens Paddle overlay or redirect
- [ ] After successful change: close modal, show success toast, refresh subscription data
- [ ] Downgrade flow: confirmation step explaining feature loss (e.g., "You'll lose access to 5 projects, Marketplace modules")

### Cancel Subscription
- [ ] "Cancel Subscription" link (subtle, not prominent) below subscription card
- [ ] Click opens confirmation dialog with:
  - Warning: "Your access will continue until {expiration_date}, then all features will be disabled"
  - Checklist of what will be lost
  - Text input requiring user to type "CANCEL" to confirm
  - "Keep My Plan" (primary) and "Cancel Subscription" (destructive) buttons
- [ ] Cancellation calls marketplace API: `POST /billing/cancel`
- [ ] After cancellation: subscription status changes to "Cancelled — Access until {date}"
- [ ] "Reactivate" button appears on cancelled subscriptions (calls `POST /billing/reactivate`)

### Invoice History
- [ ] Invoice history section below subscription card
- [ ] Table with columns: Date, Description, Amount, Status, Download
- [ ] Status badges: Paid (green), Pending (yellow), Failed (red), Refunded (blue)
- [ ] "Download PDF" link for each invoice (links to Paddle invoice PDF URL)
- [ ] Pagination if more than 10 invoices (or infinite scroll)
- [ ] Empty state: "No invoices yet"

### Payment Method
- [ ] "Manage Payment Method" button → opens Paddle Customer Portal (external link or iframe)
- [ ] Paddle Customer Portal URL fetched from marketplace API: `GET /billing/portal-url`
- [ ] Current payment method summary shown: "Visa ending in 4242" or "PayPal: user@email.com"
- [ ] If payment method is expiring soon: warning banner

### Refund Request
- [ ] "Request Refund" link in invoice row (only for Paid invoices within 30 days)
- [ ] Click opens modal with reason dropdown (Not satisfied, Bought wrong tier, Technical issues, Other)
- [ ] Free text field for additional details
- [ ] Submit sends email to support with invoice details pre-filled
- [ ] Confirmation message: "Refund request submitted. We'll respond within 2 business days."

## Technical Notes

- **Data source**: All billing data comes from marketplace API, which wraps Paddle API:
  ```typescript
  // lib/api/billing.ts
  interface Subscription {
    id: string;
    tierId: string;
    tierName: string;
    status: 'active' | 'past_due' | 'cancelled' | 'paused';
    billingCycle: 'monthly' | 'yearly';
    pricePerCycle: number;
    currency: string;
    nextBillingDate: string | null;
    startDate: string;
    cancelledAt: string | null;
    accessUntil: string | null;
    paddleSubscriptionId: string;
  }
  interface Invoice {
    id: string;
    date: string;
    description: string;
    amount: number;
    currency: string;
    status: 'paid' | 'pending' | 'failed' | 'refunded';
    pdfUrl: string | null;
    paddleTransactionId: string;
  }
  ```
- **Paddle Customer Portal**: Paddle provides a hosted portal URL. Fetch it from marketplace API and open in new tab. Alternatively, use `Paddle.Checkout.open({ settings: { type: 'overlay' } })` for inline experience.
- **Plan change flow**: Marketplace API handles the Paddle subscription update. The kaven-site just creates a checkout session for the change and Paddle handles proration.
- **File structure**:
  ```
  app/billing/
  ├── page.tsx                          # Billing page
  ├── loading.tsx                       # Loading skeleton
  components/billing/
  ├── SubscriptionCard.tsx              # Current subscription display
  ├── ChangePlanModal.tsx               # Upgrade/downgrade modal
  ├── CancelDialog.tsx                  # Cancellation confirmation
  ├── InvoiceTable.tsx                  # Invoice history table
  ├── InvoiceRow.tsx                    # Individual invoice row
  ├── PaymentMethodCard.tsx             # Payment method summary
  ├── RefundModal.tsx                   # Refund request form
  └── PlanComparisonCard.tsx            # Plan comparison in change modal
  lib/api/
  └── billing.ts                        # Billing API client
  ```
- **Optimistic updates**: After cancellation or plan change, optimistically update UI before API confirms. Rollback on error.
- **Error handling**: Payment operations are sensitive — show clear error messages, never leave user in ambiguous state.
