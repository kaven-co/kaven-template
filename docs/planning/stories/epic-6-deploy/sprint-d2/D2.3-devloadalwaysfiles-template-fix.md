# D2.3 — `devLoadAlwaysFiles` Docs no Template

> **Epic:** 6 (Deploy) | **Sprint:** D2 | **Agent:** @kaven-devops
> **Priority:** P2
> **Depends on:** C3.1 (`kaven init` AIOX Environment Bootstrap)

---

## Contexto

O `AIOX-KAVEN-INTEGRATION.md` (GAP-8, Seção 2.5) documenta um problema descoberto durante a instalação do AIOX v5.0.3 no kaven-framework em 2026-03-11:

O arquivo `.aiox-core/core-config.yaml` instalado pelo AIOX v5 declara um campo `devLoadAlwaysFiles` que lista arquivos a serem carregados automaticamente no início de cada sessão de agent. Os paths esperados são:

```
docs/architecture/coding-standards.md
docs/architecture/tech-stack.md
docs/architecture/source-tree.md
```

**Esses arquivos não existem** no `kaven-template` nem em projetos criados via `kaven init`. O kaven-squad possui equivalentes (`tech-stack.md`, `source-tree.md`, `coding-standards.md`) em `squads/kaven-squad/config/`, mas o AIOX não sabe disso — ele procura nos paths padrão esperados pelo core-config.

**Consequência:** Agents como `@dev`, `@qa`, `@architect`, `@pm`, `@sm` iniciam cada sessão sem carregar as docs de contexto do framework. Isso leva a:
- Agents tomando decisões que ignoram o 10-layer middleware do Kaven
- Respostas que desconhecem a estrutura `apps/api`, `apps/admin`, `apps/tenant`
- `@qa` escrevendo testes sem awareness dos patterns de test isolation do Kaven
- Perda de tempo resolvendo problemas já documentados no kaven-squad config

Esta story garante que projetos criados via `kaven init` tenham os 3 arquivos de docs no path esperado pelo AIOX, eliminando os warnings de "file not found" e garantindo que agents comecem sessões com contexto completo.

## User Story

**As a** developer who ran `kaven init my-saas --with-squad`,
**I want to** have the framework architecture docs automatically available at the paths AIOX expects,
**So that** all agents start their sessions with complete Kaven context without any "file not found" warnings.

## Technical Notes

### Abordagem Escolhida

O `AIOX-KAVEN-INTEGRATION.md` propõe 3 opções. A **Opção 1** foi escolhida como solução imediata:

> Criar `docs/architecture/` no `kaven-template` — os 3 arquivos gerados durante `kaven init` com template vars substituídas.

A **Opção 2** (atualizar `core-config.yaml`) é a solução de longo prazo e pode ser implementada em sprint posterior como melhoria.

#### `docs/architecture/coding-standards.md`

```markdown
# Coding Standards — {{PROJECT_NAME}}

> Generated by `kaven init` — Kaven Framework v1.0.0-rc1

## Git & Commits
- **Branch strategy:** `feat/`, `fix/`, `chore/`, `refactor/`, `test/`, `docs/`
- **Commits:** Conventional Commits, ≤72 chars, inglês
- **Never:** push --force em branches compartilhadas, commitar secrets
- **Always:** commitar antes de mover para próxima task

## TypeScript
- **Strict mode:** `"strict": true` em todos os tsconfig
- **No `any`:** usar `unknown` quando tipo não é conhecido
- **Zod validation:** todos os inputs externos (API body, env vars) validados com Zod
- **TypeBox:** schemas Fastify usam TypeBox para JSON Schema + inferência TypeScript

## API (Fastify)
- **Route structure:** `src/modules/<module>/routes/<module>.routes.ts`
- **Service layer:** lógica de negócio em `src/modules/<module>/services/<module>.service.ts`
- **Middleware hooks:** registrar via `fastify.addHook('preHandler', ...)`
- **Response format:** `{ data: T }` (success) | `{ error: string, code: string }` (failure)
- **Pagination:** `{ data: T[], meta: { total, page, limit } }`
- **tenantId:** OBRIGATÓRIO em todas as queries de dados — nunca buscar sem tenant context

## Database (Prisma)
- **Soft delete:** campo `deletedAt` em todos os models (nunca delete físico)
- **Indexes:** composite index em `(tenantId, <primary-filter>)` em todos os models
- **Migrations:** uma migration por feature/story — nomes descritivos
- **RLS:** `prisma-rls.ts` intercepta todas as queries — não bypassar sem razão explícita
- **Extended schema:** novos models em `schema.extended.prisma`, não no `schema.base.prisma`

## Testing
- **Framework:** Vitest + Supertest
- **Coverage target:** 80%+ (statement)
- **Test isolation:** cada teste usa transaction rollback ou memfs para filesystem
- **Multi-tenant tests:** sempre testar que tenant A não acessa dados do tenant B
- **Security tests:** IDOR tests para todos os endpoints com IDs externos

## Error Handling
- **Never empty catch:** sempre logar o erro (`request.log.error({ err }, 'message')`)
- **HTTP errors:** usar `fastify.httpErrors.<statusCode>(message)` (fastify-sensible)
- **Async errors:** sempre `await` ou `.catch()` em Promises
```

### Substituição de Template Vars em `kaven init`

```typescript
// kaven-cli: src/commands/init/config-replacer.ts
// Adicionar ao replacePlaceholders():

const TEMPLATE_VARS = { '{{PROJECT_NAME}}': config.name };

const ARCH_DOCS = [
  'docs/architecture/coding-standards.md',
  'docs/architecture/tech-stack.md',
  'docs/architecture/source-tree.md',
];

for (const docPath of ARCH_DOCS) {
  const fullPath = path.join(projectDir, docPath);
  if (fs.existsSync(fullPath)) {
    let content = fs.readFileSync(fullPath, 'utf-8');
    for (const [placeholder, value] of Object.entries(TEMPLATE_VARS)) {
      content = content.replace(new RegExp(placeholder, 'g'), value);
    }
    fs.writeFileSync(fullPath, content);
  }
}
```
