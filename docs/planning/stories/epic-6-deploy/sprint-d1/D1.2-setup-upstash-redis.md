# D1.2 — Setup Upstash Redis — Production Cache & Rate Limiting

> **Epic:** 6 (Deploy & Launch) | **Sprint:** D1 | **Agent:** @devops (Deploy/Gage)
> **Priority:** P0 — Marketplace API usa Redis para rate limiting e cache de sessões
> **Depends on:** Nenhum

---

## Contexto

O kaven-marketplace usa Redis para:
- Rate limiting (60 req/min por IP, 100 req/min por token)
- Cache de JWT validation results (TTL 15min)
- Session storage para device code flow

Upstash foi escolhido por ser serverless Redis (paga por request), sem servidores para gerenciar, e tem free tier de 10K commands/dia (suficiente para launch).

## User Story

**Como** DevOps do Kaven,
**Quero** criar um database Redis no Upstash para produção,
**Para que** o Marketplace API tenha rate limiting e caching funcionando.

## Acceptance Criteria

- [ ] Conta Upstash criada em upstash.com
- [ ] Database Redis criada (região us-east-1 para latência com Railway)
- [ ] Conexão testada: `redis-cli -u $REDIS_URL ping` retorna `PONG`
- [ ] TLS/SSL ativado na conexão
- [ ] `REDIS_URL` salvo no password manager
- [ ] Env vars documentadas para marketplace

## Technical Notes

```bash
# Testar conexão
redis-cli -u "rediss://default:password@xxx.upstash.io:6380" ping
# Deve retornar: PONG

# Ou via Node.js
node -e "
const { Redis } = require('@upstash/redis');
const r = new Redis({ url: process.env.REDIS_URL, token: process.env.REDIS_TOKEN });
r.ping().then(console.log);
"
```
