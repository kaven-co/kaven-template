## Contexto

O script `packages/database/scripts/stripe-orphan-sweep.ts` existe e é prod-safe após F2.4.3.3 (age-gate 1h, delete idempotente, audit sidecar, live-mode gate). Hoje ele é invocável apenas manualmente via `pnpm --filter @kaven/database stripe:sweep`.

Esta story operacionaliza o sweep: cron schedule em prod + integração com `AuditEvent` DB para que logs sejam queryáveis no painel admin, não apenas arquivos em disco.

---

## Escopo

### 1. Cloud Run Job (GCP)

- Container image: mesma do `@kaven/database` package + dep Stripe SDK.
- Schedule: **semanal** (segunda 03:00 UTC), via Cloud Scheduler.
- Env vars: `STRIPE_SECRET_KEY` (sk_live_*** em prod, sk_test em staging), `DATABASE_URL`.
- `--delete` não é passado por default — apenas dry-run semanal. Pipeline de escalonamento: alerta se `orphans > 10` em dry-run semanal → operador investiga → executa manualmente `--delete + STRIPE_SWEEP_CONFIRM=yes + ALLOW_STRIPE_LIVE_DELETE=yes`.

### 2. AuditEvent integration

- Ao final do sweep, emitir 1 `AuditEvent` por execução:
  - `type = 'stripe.sweep.completed'`
  - `actor = 'system'` (ou `job:stripe-orphan-sweep`)
  - `payload = { dryRun, confirmed, keyMode, totalKavenCustomers, deleted, orphans, dangling, skippedByAge, skippedAlreadyDeleted, errors, startedAt, finishedAt }`
- Se `orphans.length >= threshold` (default 10), também emitir `type = 'stripe.sweep.alert'` com severity `warning`.
- Persistir o sidecar JSON também em Cloud Storage bucket (opcional, backup de 30d) — decidir em revisão.

### 3. Alerting

- Datadog/Cloud Monitoring alerta se:
  - `errors.length > 0` em qualquer execução.
  - `orphans.length > 10` em duas execuções consecutivas (persistência de problema).
  - Job não executou em > 10 dias (scheduler caiu).

### 4. Runbook ops (`docs/runbooks/stripe-sweep.md`)

Documentar:
- Como invocar manualmente (staging vs prod, dry-run vs destrutivo).
- Triagem de alertas (orphan → investigar signup logs; dangling → verificar se tenant foi hard-deleted; errors → escalar).
- Rollback: Stripe Customer deletion é IRREVERSÍVEL; não há rollback. Evitar via age-gate + dupla confirmação.
