# F1.2 — Plan Change Webhook Handler

> **Epic:** 4 (Framework — Upgrade Flow & RC1) | **Sprint:** F1 | **Agent:** @dev
> **Priority:** P0
> **Depends on:** M2.4

---

## Contexto

Quando um tenant completa um pagamento via Paddle (iniciado pelo F1.1), o Paddle envia webhooks para o kaven-framework backend. O webhook handler precisa processar eventos de subscription (criacao, atualizacao, cancelamento), mapear price IDs do Paddle para Plan IDs do Kaven, atualizar o Subscription e PlanFeature assignments no banco, e recalcular usage limits.

O kaven-framework ja tem modelos Prisma para Subscription, Plan, PlanFeature, e o middleware `requireFeature()` que checa limites. O que falta e o endpoint webhook que conecta Paddle a essas entidades.

A idempotencia e critica: Paddle pode reenviar webhooks, e o handler deve processar cada evento exatamente uma vez.

---

## User Story

**Como** sistema do Kaven (backend),
**Quero** processar webhooks do Paddle quando subscriptions mudam,
**Para que** o plano do tenant seja atualizado automaticamente, features sejam habilitadas/desabilitadas, e limites recalculados sem intervencao manual.

## Acceptance Criteria

### Webhook Endpoint
- [ ] Fastify route: `POST /webhooks/paddle` registered in the API
- [ ] Endpoint is public (no auth middleware) but validates Paddle signature
- [ ] Paddle webhook signature verified using `Paddle-Signature` header and webhook secret
- [ ] Invalid signature returns 401 and logs warning
- [ ] Valid webhook returns 200 immediately (before processing) to avoid Paddle timeout retries
- [ ] Processing happens asynchronously after response is sent

### Event: subscription_created
- [ ] When Paddle sends `subscription.created` event:
  - Extract: subscription ID, customer ID, price ID, status, billing dates
  - Map Paddle `price_id` to Kaven `planId` using config mapping table
  - Find or create tenant from customer metadata (tenantId passed in checkout)
  - Create `Subscription` record in DB:
    ```
    { paddleSubscriptionId, tenantId, planId, status: 'ACTIVE',
      billingCycle, currentPeriodStart, currentPeriodEnd, paddleCustomerId }
    ```
  - Update tenant's `currentPlanId` to new plan
  - Create/update `PlanFeature` assignments based on plan's feature set
  - Log event in audit log

### Event: subscription_updated
- [ ] When Paddle sends `subscription.updated` event:
  - Find existing `Subscription` by `paddleSubscriptionId`
  - If price changed (plan change): update `planId`, recalculate features
  - Update billing dates: `currentPeriodStart`, `currentPeriodEnd`
  - Update status if changed
  - If downgrade: schedule feature adjustment for end of current period (grace period)
  - Log event in audit log

### Event: subscription_cancelled
- [ ] When Paddle sends `subscription.cancelled` event:
  - Find existing `Subscription` by `paddleSubscriptionId`
  - Set status to `CANCELLED`
  - Set `cancellationDate` and `accessUntil` (end of current billing period)
  - Do NOT remove features immediately — tenant keeps access until `accessUntil`
  - Schedule a job/cron to revoke features after `accessUntil` date
  - Send notification to tenant admin: "Your subscription has been cancelled. Access continues until {date}."
  - Log event in audit log

### Event: subscription_payment_failed
- [ ] When Paddle sends `transaction.payment_failed` event:
  - Find existing `Subscription` by `paddleSubscriptionId`
  - Set status to `PAST_DUE`
  - Send notification to tenant admin: "Payment failed. Please update your payment method."
  - After 3 consecutive failures (check retry count): mark as `SUSPENDED`
  - Log event in audit log

### Paddle-to-Kaven Mapping
- [ ] Config file maps Paddle price IDs to Kaven plan IDs:
  ```typescript
  // config/paddle-mapping.ts
  const PADDLE_PLAN_MAP: Record<string, string> = {
    'pri_starter_monthly': 'plan-starter-id',
    'pri_starter_yearly': 'plan-starter-id',
    'pri_complete_monthly': 'plan-complete-id',
    'pri_complete_yearly': 'plan-complete-id',
    'pri_pro_monthly': 'plan-pro-id',
    'pri_pro_yearly': 'plan-pro-id',
  };
  ```
- [ ] Unknown price ID: log error, store raw event, return 200 (don't block Paddle)

### Idempotency
- [ ] Store processed event IDs in a `WebhookEvent` table: `{ paddleEventId, eventType, processedAt, payload }`
- [ ] Before processing: check if `paddleEventId` already exists
- [ ] If already processed: return 200 without re-processing
- [ ] If new: process and insert event record in same transaction

### Feature Recalculation
- [ ] After plan change: call `PlanFeatureService.recalculateForTenant(tenantId, newPlanId)`
- [ ] Recalculation updates all 40+ feature capabilities based on new plan's limits
- [ ] Usage counters preserved (not reset on plan change)
- [ ] If downgrading: check if current usage exceeds new limits, flag as "over limit" but don't delete data

### Notifications
- [ ] Use existing notification system to inform tenant admin about:
  - Subscription activated: "Welcome to {Plan}! Your features have been updated."
  - Plan changed: "Your plan has been changed to {Plan}."
  - Subscription cancelled: "Your subscription has been cancelled. Access continues until {date}."
  - Payment failed: "Payment failed. Please update your payment method to avoid service interruption."

## Technical Notes

- **Paddle webhook signature verification**:
  ```typescript
  import crypto from 'crypto';

  function verifyPaddleWebhook(rawBody: string, signature: string, secret: string): boolean {
    const ts = signature.split(';').find(p => p.startsWith('ts='))?.split('=')[1];
    const h1 = signature.split(';').find(p => p.startsWith('h1='))?.split('=')[1];
    const payload = `${ts}:${rawBody}`;
    const expectedSignature = crypto.createHmac('sha256', secret).update(payload).digest('hex');
    return crypto.timingSafeEqual(Buffer.from(h1!), Buffer.from(expectedSignature));
  }
  ```
- **Async processing pattern**: Fastify supports `reply.send()` before async work. Use this to return 200 immediately:
  ```typescript
  fastify.post('/webhooks/paddle', async (request, reply) => {
    // Verify signature synchronously
    if (!verifySignature(request)) return reply.status(401).send();
    // Send 200 immediately
    reply.status(200).send({ received: true });
    // Process asynchronously
    setImmediate(() => processWebhookEvent(request.body));
  });
  ```
- **Prisma models involved**: `Subscription`, `Plan`, `PlanFeature`, `Tenant`, `WebhookEvent` (new), `AuditLog`
- **WebhookEvent schema** (new Prisma model):
  ```prisma
  model WebhookEvent {
    id              String   @id @default(cuid())
    source          String   // "paddle"
    externalEventId String   @unique // Paddle event ID
    eventType       String   // "subscription.created", etc.
    payload         Json
    processedAt     DateTime @default(now())
    status          String   @default("processed") // "processed", "failed", "skipped"
    errorMessage    String?
    createdAt       DateTime @default(now())
  }
  ```
