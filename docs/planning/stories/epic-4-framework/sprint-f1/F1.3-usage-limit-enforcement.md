# F1.3 — Usage Limit Enforcement

> **Epic:** 4 (Framework — Upgrade Flow & RC1) | **Sprint:** F1 | **Agent:** @dev
> **Priority:** P1
> **Depends on:** F1.1

---

## Contexto

O kaven-framework ja tem o sistema de feature flags com 40+ capabilities, `requireFeature()` middleware, `FeatureLimitCard`, `PlanUsageSummary`, e `UsageTrackingService` com real counters. O que falta e a "ultima milha": conectar o enforcement de limites com a UI de upgrade. Quando um tenant atinge 80% do quota, deve ver um warning. Ao atingir 100%, a acao e bloqueada e um prompt de upgrade aparece com comparacao de planos e link direto para o Paddle checkout (via F1.1).

Esta story transforma os limites de "numeros no banco" em UX real que guia o usuario para o upgrade sem ser agressivo.

---

## User Story

**Como** tenant admin que esta chegando perto dos limites do meu plano,
**Quero** ser avisado progressivamente e ter um caminho facil para upgrade,
**Para que** eu nao seja surpreendido por bloqueios e possa crescer meu uso de forma natural.

## Acceptance Criteria

### Soft Warning (80% Threshold)
- [ ] When a usage counter reaches 80% of the plan limit, a warning banner appears
- [ ] Banner text: "You've used {current}/{limit} of your {resource} quota. Consider upgrading for more capacity."
- [ ] Banner is dismissible (per session, reappears on next login)
- [ ] Banner color: yellow/amber warning style
- [ ] Banner includes "Upgrade" link that scrolls to or navigates to the plans page
- [ ] Warning triggers for all trackable resources: users, tenants, projects, API calls, storage

### Hard Block (100% Threshold)
- [ ] When usage reaches 100% of limit, the action is blocked
- [ ] Block happens at the API level: `requireFeature()` middleware returns 403 with upgrade info
- [ ] API response includes: `{ error: "LIMIT_REACHED", resource: "users", current: 10, limit: 10, suggestedPlan: "complete" }`
- [ ] Frontend catches 403 with `LIMIT_REACHED` code and shows upgrade prompt modal
- [ ] Modal content:
  - "You've reached your {resource} limit"
  - Current plan summary with usage bar (full/red)
  - Recommended plan with price and key benefit
  - Price difference shown: "Upgrade for +${difference}/month"
  - "Upgrade Now" button → Paddle checkout (reuses F1.1 flow)
  - "Maybe Later" dismiss button
- [ ] Blocked action examples: creating a new user beyond limit, adding a new project, creating a tenant

### Upgrade Prompt Component
- [ ] Reusable `UpgradePrompt` component that can be triggered from any feature limit
- [ ] Component accepts: `resource`, `currentUsage`, `limit`, `suggestedPlan`
- [ ] Shows plan comparison mini-table: current vs suggested plan for the specific resource
- [ ] "Upgrade Now" button wired to Paddle checkout via F1.1 UpgradeButton
- [ ] After successful upgrade: automatically dismiss prompt, refresh usage data, show success toast
- [ ] Component also usable inline (not just modal) — e.g., embedded in FeatureLimitCard

### Usage Bars & Indicators
- [ ] Existing `FeatureLimitCard` enhanced with color-coded progress bar:
  - 0-59%: green
  - 60-79%: blue
  - 80-94%: yellow/amber with pulse animation
  - 95-99%: orange
  - 100%: red with "Limit reached" label
- [ ] `PlanUsageSummary` component enhanced to show all trackable resources with bars
- [ ] Dashboard widget: usage summary card showing top 3 resources closest to limits
- [ ] Tooltip on hover showing exact numbers: "8 of 10 users (80%)"

### Post-Upgrade Refresh
- [ ] After successful Paddle checkout (F1.1 `checkout.completed` event):
  - Invalidate all usage cache
  - Refetch plan data and feature limits from API
  - Remove warning banners and blocks immediately
  - Show success message: "Plan upgraded to {Plan}! Your new limits are now active."
  - Animate usage bars updating to new limits (old usage / new higher limit)
- [ ] Refresh is automatic — no manual page reload needed

### Edge Cases
- [ ] If tenant is already on the highest plan (Pro) and hits limits: show "Contact us for Enterprise" instead of upgrade
- [ ] If tenant has cancelled subscription: show "Reactivate your subscription to continue" instead of upgrade
- [ ] Grace period for downgrades: if tenant is over new plan's limits after downgrade, show warning but don't delete data
- [ ] Concurrent usage: if two users trigger the limit simultaneously, both see the block but only one needs to upgrade

## Technical Notes

- **Existing services to leverage**:
  - `UsageTrackingService` — already tracks counters per tenant per resource
  - `requireFeature()` middleware — already checks feature flags, needs enhancement for soft/hard limits
  - `FeatureLimitCard` — existing component, needs color-coded progress bar
  - `PlanUsageSummary` — existing component, needs resource-level bars
- **Enhanced `requireFeature()` middleware**:
  ```typescript
  // Enhanced to return limit info on 403
  async function requireFeature(feature: string) {
    const usage = await UsageTrackingService.getUsage(tenantId, feature);
    const limit = await PlanFeatureService.getLimit(tenantId, feature);
    if (usage >= limit) {
      return reply.status(403).send({
        error: 'LIMIT_REACHED',
        resource: feature,
        current: usage,
        limit: limit,
        suggestedPlan: PlanService.getSuggestedUpgrade(currentPlanId, feature),
      });
    }
    // Also return warning header if > 80%
    if (usage / limit >= 0.8) {
      reply.header('X-Usage-Warning', JSON.stringify({ resource: feature, current: usage, limit }));
    }
  }
  ```
- **Warning banner via response headers**: API returns `X-Usage-Warning` header when usage > 80%. Frontend interceptor reads this header and shows banner. This avoids polling.

- **API interceptor** (for X-Usage-Warning header):
  ```typescript
  // lib/api/interceptor.ts
  function handleResponse(response: Response) {
    const warning = response.headers.get('X-Usage-Warning');
    if (warning) {
      const { resource, current, limit } = JSON.parse(warning);
      showUsageWarning(resource, current, limit);
    }
    if (response.status === 403) {
      const data = await response.json();
      if (data.error === 'LIMIT_REACHED') {
        showUpgradePrompt(data);
        throw new LimitReachedError(data);
      }
    }
  }
  ```
- **Color gradient utility**:
  ```typescript
  function getUsageColor(percentage: number): string {
    if (percentage >= 100) return 'red';
    if (percentage >= 95) return 'orange';
    if (percentage >= 80) return 'amber';
    if (percentage >= 60) return 'blue';
    return 'green';
  }
  ```
