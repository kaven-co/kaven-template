## Contexto

QA sweep em 2026-04-18 (architecture-strategist + data-integrity-guardian + security-sentinel) identificou **5 blockers convergentes** e **3 HIGH findings de segurança** em F2.4.1→F2.4.3. Nenhum é FAIL isolado, mas juntos formam um funil de corrupção silenciosa que cresce a cada signup pós-F2.4.3 e vira incidente quando:

1. F2.4.5 tentar `ALTER TABLE tenants ALTER COLUMN billing_account_id SET NOT NULL`
2. EPIC-007 (kaven-site migration) trouxer volume real a produção
3. F2.4.4 expor endpoints públicos de `BillingAccount` (amplifica IDOR existente em `/api/checkout`)

Volume atual (1 tenant) é a única razão pela qual ainda não é incidente. Fix agora é cirúrgico; depois vira recovery com data loss.

---

## Escopo

### Blockers convergentes (flagrados por 2+ revisores)

**B1. Role mismatch signup × backfill** (Arch + Data-Integrity + Security)
- `auth.service.register` cria `role: "USER"` hardcoded.
- `f2-4-backfill` exige `role: 'TENANT_ADMIN'` para resolver `BA.ownerUserId`.
- Tenants criados via signup F2.4.3 **não são reparáveis** pelo backfill se BA create falhar → F2.4.5 NOT NULL aborta.
- **Fix:** quando user cria o próprio tenant (modo camaleão, `!data.tenantSlug`), role = `TENANT_ADMIN`. Senão mantém `USER`.

**B2. Signup BA non-blocking + sem transação** (Arch + Data-Integrity + Security)
- `try/catch` engole falha de BA create com apenas `secureLog.warn` → tenant órfão permanente.
- 6 operações (tenant, stripe, update tenant, user, BA, update tenant) em conexões independentes → 4 janelas de orfanização.
- `stripe.customers.create` sem `idempotencyKey` → duplica Stripe Customer em retry de rede.
- **Fix:** envolver tenant + user + BA + links em `prisma.$transaction`. Falha de BA → throw (não warn). Stripe com `idempotencyKey`.

**B3. Rollback SQL deleta BAs legítimas** (Arch + Data-Integrity)
- `DELETE FROM billing_accounts WHERE created_at >= '2026-04-18'` em `f2-4-rollback.sql` apaga BAs criadas por signup F2.4.3 que legitimamente devem permanecer.
- **Fix:** adicionar coluna `source` em `billing_accounts` (`'signup'` | `'backfill_f2_4_2'`). Backfill grava `'backfill_f2_4_2'`. Rollback filtra `WHERE source = 'backfill_f2_4_2'`.

**B4. Race condition backfill × signup concorrente** (Arch + Data-Integrity)
- Backfill lê snapshot de tenants em `main`, processa em transações individuais. Se signup cria BA entre SELECT e UPDATE → backfill cria 2ª BA + orfana a 1ª + possível UNIQUE violation em `stripeCustomerId`.
- **Fix:** no backfill, `UPDATE ... WHERE id = $tenant AND billing_account_id IS NULL` + check rowCount → throw se race detectada.

**B5. Dois source-of-truth para `stripeCustomerId`** (Arch + Data-Integrity + Security)
- Signup escreve em AMBOS `Tenant.stripeCustomerId` e `BillingAccount.stripeCustomerId`.
- Checkout escreve em UM dos dois dependendo de `billingAccountId`.
- Divergência silenciosa possível entre os dois campos.
- **Fix:** ADR definindo regra. Quando `billingAccountId IS NOT NULL` → BA é single-source para writes. Signup para de gravar `Tenant.stripeCustomerId` (só BA). Campo Tenant permanece apenas para tenants legacy pré-F2.4.

### Security HIGH findings (amplificam F2.4.4)

**H-1. IDOR em `/api/checkout/create-session`**
- Guard `if (requestingUser?.tenantId && ...)` passa se `tenantId = null` (SUPER_ADMIN, legacy users, F2.4.6 billing-owners).
- Atacante autenticado manda qualquer `tenantId` no body → checkout autorizado.
- **Fix:** deny-by-default. Exceções explícitas: SUPER_ADMIN via role, ownership via BA (para F2.4.4).

**H-2. Billing hijack derivado de H-1**
- Se BA da vítima tem `stripeCustomerId = null`, checkout cria Stripe Customer com `email = requestingUser.email` (atacante) → persiste em `BillingAccount.stripeCustomerId` da vítima → `idempotencyKey` cimenta hijack.
- **Fix:** email do Stripe Customer lido de `BA.owner.email`, não do requester. Escrita em `BA.stripeCustomerId` via `updateMany where: { stripeCustomerId: null }` (write-once).

**H-3. PII leak em logs**
- `secureLog` não tem `stripeCustomerId`/`ownerUserId` em `SENSITIVE_KEYS` → valores `cus_*` (18 chars) passam o threshold `length > 20` sem redação.
- `f2-4-backfill.ts` e `f2-4-pre-migration-audit.ts` usam `console.log` direto com email + Stripe ID.
- Relatório markdown do audit (`docs/migrations/f2-4-audit-{date}.md`) pode ser comitado com PII.
- **Fix:** expandir `SENSITIVE_KEYS`. Threshold redation ≤ 14 chars. Scripts usam função de mascaramento. Audit grava PII detalhado em JSON separado gitignored.

## Acceptance Criteria

### AC1 — Signup atomic + BA blocking
- [ ] `register` envolve tenant + user + BA + links em `prisma.$transaction`.
- [ ] Falha de BA create lança exception (não apenas warn) → rollback da transação.
- [ ] `stripe.customers.create` no signup com `idempotencyKey: signup-customer-${tenantSlug}`.
- [ ] Quando criador é owner de novo tenant (camaleão), `role = TENANT_ADMIN`.
- [ ] Teste de integração: falha injetada em `billingAccount.create` → tenant + user NÃO persistem.
- [ ] Teste de integração: retry do signup com mesmo slug → não cria Stripe Customer duplicado.

### AC2 — Backfill race-safe
- [ ] `UPDATE "Tenant"` no backfill com `AND billing_account_id IS NULL`.
- [ ] Check `rowCount === 0` → throw erro dentro da transação + registro em `result.errors`.
- [ ] Teste unitário com mock Prisma demonstrando race detection.

### AC3 — Rollback distingue origem
- [ ] Migration adiciona coluna `source TEXT NOT NULL DEFAULT 'signup'` em `billing_accounts`.
- [ ] `f2-4-backfill.ts` grava `source: 'backfill_f2_4_2'` em BAs criadas.
- [ ] `f2-4-rollback.sql` filtra por `source = 'backfill_f2_4_2'` (não por data).
- [ ] Teste manual documentado: rollback após signup → signup BAs preservadas.

### AC4 — Single-write para stripeCustomerId
- [ ] Signup quando cria BA **não** escreve em `Tenant.stripeCustomerId` (só em BA).
- [ ] Para tenants pré-F2.4 sem BA, `Tenant.stripeCustomerId` permanece funcional (backward-compat).
- [ ] ADR em `docs/adrs/ADR-{NNNN}-billing-account-source-of-truth.md`.

### AC5 — Checkout AuthZ deny-by-default
- [ ] Guard: `if (!requestingUser?.tenantId || requestingUser.tenantId !== tenantId) { deny UNLESS SUPER_ADMIN }`.
- [ ] Email do Stripe Customer derivado de `BA.owner.email` (ou tenant admin se sem BA) — nunca do requester.
- [ ] Write em `BA.stripeCustomerId` via `updateMany` com `stripeCustomerId: null` (write-once).
- [ ] Teste E2E: user com `tenantId=null` tentando checkout de outro tenant → 403.
- [ ] Teste E2E: user com `tenantId=X` tentando checkout de tenant `Y` → 403.

### AC6 — Log sanitization
- [ ] `secure-logger.ts SENSITIVE_KEYS` inclui: `stripecustomerid`, `stripe_customer_id`, `customer_id`, `owneruserid`, `owner_user_id`, `email`, `billingaccountid`.
- [ ] Threshold de redação reduzido de `> 20` para `> 14` chars.
- [ ] `f2-4-backfill.ts` usa função `maskEmail()` e `maskStripeId()` em logs.
- [ ] `f2-4-pre-migration-audit.ts` grava detalhes sensíveis em `docs/migrations/f2-4-audit-{date}.json` (gitignored), markdown fica apenas com contagens.
- [ ] `.gitignore` inclui `docs/migrations/*-audit-*.json`.
- [ ] Teste unitário do secureLog demonstrando redação de `cus_`.

### AC7 — CI verde + testes
- [ ] `pnpm typecheck` em todos os workspaces.
- [ ] `pnpm test` no `apps/api` (suite atual deve continuar passando + novos testes de regressão).
- [ ] `pnpm lint` sem novos warnings.
- [ ] `scripts/hooks/pre-pr.sh` verde.

## Riscos

- **Baixo**: mudança de `role: USER → TENANT_ADMIN` em signup pode quebrar algum teste ou consumer que assume `USER`. Mitigação: grep + teste de regressão em RBAC.
- **Baixo**: envolver em transação pode aumentar lock contention no signup. Mitigação: Stripe permanece fora da transação (idempotente via key).
- **Médio**: nova coluna `source` exige migration + seed para BAs existentes (= 1 BA em dev). Script preenche BAs existentes com `'backfill_f2_4_2'` se `created_at > 2026-04-18` senão `'signup'` (heurística; não há BAs legacy).
