## Architect Review (2026-04-18, Orion)

Design original validado com **6 ajustes** consolidados abaixo. Pontos críticos auditados:

| Tema | Decisão final |
|---|---|
| Cardinalidade BA ↔ Subscription | **1:N** (BA owns N Subscriptions). MVP usa 1:1 mas FK fica em `Subscription.billingAccountId`, abrindo extensão futura sem schema change. |
| Cardinalidade BA ↔ Tenant | **1:N** (core feature: 1 BA owna até `MAX_TENANTS` workspaces). |
| Quem é o billing holder na v1 | **Single `ownerUserId`** em BA. Multi-owner (`BillingAccountMember` com role) fica em **F2.4.6 deferida** (não bloqueia ativação de MAX_TENANTS). |
| Generalização da quota engine | **Adicionar `Feature.scope` enum** (`TENANT` \| `BILLING_ACCOUNT`). `EntitlementService.resolveQuotaCurrentUsage` passa a despachar baseado em scope. Mantém retrocompat de `MAX_PROJECTS` (scope=TENANT) e habilita `MAX_TENANTS` (scope=BILLING_ACCOUNT) sem refactor cross-cutting. |
| Pre-migration audit script | **OBRIGATÓRIO** (nova fase 0). Detecta tenants sem owner user, subscriptions órfãs, stripe customers duplicados, etc. Story F2.4.0 abaixo. |
| Subscription.tenantId após Phase 3 | **Manter como nullable** (não dropar) por 1 sprint pós-stabilização. Reduz blast radius de breakage em código legacy de checkout. Drop fica em story futura `F2.5 Subscription cleanup`. |
| Plano downgrade com tenants excedendo limite | **Soft-block**: tenants existentes ficam acessíveis (read-only ou normal — owner decide), mas nenhum novo tenant pode ser criado até voltar para o plano com quota suficiente. Documentado como UX deliberada. |

### Riscos críticos identificados pelo review

1. **`Tenant.stripeCustomerId` está hoje como cast `(tenant as any)`** porque o campo NÃO está em `schema.base.prisma` (foi adicionado por PR #114). O backfill F2.4.2 precisa ler dessa coluna ANTES dela ser removida em F2.4.5. Ordem de execução é estritamente sequencial.

2. **Signup flow** (criação de novo usuário/tenant) **MUDA** — hoje cria User → Tenant → Subscription. Pós-F2.4 vira User → BillingAccount → Tenant → Subscription. Sub-story F2.4.3 trata isso explicitamente.

3. **O endpoint `POST /api/tenants` (SUPER_ADMIN-only) NÃO é deprecado** nesta story — fica para uso interno de plataforma (provisionar tenants para enterprise customers, debug, etc.). Deprecation real fica para após estabilização.

4. **Authorization no novo endpoint**: `POST /api/billing-accounts/:id/tenants` requer caller ser `ownerUserId` da BA OU ter role SUPER_ADMIN. Verificação implementada via novo `requireBillingAccountOwner` middleware (não reutiliza `requireRole` porque a relação é por FK, não por role estática).

## Contexto (preservado do draft)

O sprint F2 entregou infra completa de enforcement de quota por plano (F2.1 MAX_TENANTS + F2.2 MAX_PROJECTS + F2.3 hardening). F2.2 está **ativo em produção** via `requireFeature('MAX_PROJECTS')` em `POST /api/v1/projects`. F2.1 **não está ativo** porque:

1. O único endpoint de criação de tenant (`POST /api/tenants`) é **SUPER_ADMIN-only**, e SUPER_ADMIN bypassa quotas por design (`feature-guard.middleware.ts`).
2. O modelo de dados atual é **1 Tenant → 1 Subscription** (relacionamento 1:1 via `Subscription.tenantId @unique`). Não existe um conceito de "billing account" que possa owner N tenants.
3. Com o modelo atual, `MAX_TENANTS=3` por plano Pro não tem como ser interpretado — o plano está na Subscription, que já pertence a um único Tenant.

A infra de F2.1 (seed de features + `EntitlementService.resolveQuotaCurrentUsage` contando tenants ativos) **permanece pronta** para ser plugada no endpoint certo assim que o modelo de ownership existir.

**Hoje** (`apps/api/src/modules/subscriptions/services/entitlement.service.ts:174`):
```ts
if (featureCode === 'MAX_TENANTS') {
  return prisma.tenant.count({ where: { deletedAt: null } }); // ⚠️ contagem GLOBAL — wrong
}
```

**Depois de F2.4**:
```ts
if (feature.scope === 'BILLING_ACCOUNT') {
  return prisma.tenant.count({ where: { billingAccountId, deletedAt: null } });
}
```

## User Story

**Como** customer Pro do Kaven,
**Quero** poder criar e gerenciar até 3 workspaces (tenants) distintos sob uma única assinatura,
**Para que** eu possa separar contextos (ex: agências com múltiplos clientes, grupos empresariais com filiais) sem pagar N assinaturas.

**Como** engenheiro de plataforma,
**Quero** um modelo canônico de "BillingAccount owner-of Tenants" com enforcement automático,
**Para que** o campo `MAX_TENANTS` dos planos deixe de ser vaporware e passe a regular criação de workspaces de verdade.

## Design de Arquitetura (✅ approved by @architect)

### Entidade nova: `BillingAccount`

```prisma
model BillingAccount {
  id               String   @id @default(uuid())

  /// User dono/billing admin desta BA. Single-owner na v1.
  /// Multi-owner via BillingAccountMember entra em F2.4.6 (deferred).
  ownerUserId      String   @map("owner_user_id")

  /// Stripe Customer ID (movido de Subscription.stripeCustomerId após F2.4.2).
  /// Mantém @unique para preservar invariante 1:1 com Stripe.
  stripeCustomerId String?  @unique @map("stripe_customer_id")

  /// Display name / razão social (opcional v1, obrigatório futuro).
  name             String?

  createdAt        DateTime @default(now()) @map("created_at")
  updatedAt        DateTime @updatedAt @map("updated_at")
  deletedAt        DateTime? @map("deleted_at")

  // Relations
  owner            User           @relation(fields: [ownerUserId], references: [id], onDelete: Restrict)
  tenants          Tenant[]
  subscriptions    Subscription[]

  @@index([ownerUserId])
  @@index([stripeCustomerId])
  @@map("billing_accounts")
}
```

### Mudanças em entidades existentes

```prisma
model Tenant {
  // ... existing fields ...

  /// Phase 1: nullable. Phase 3: NOT NULL.
  billingAccountId String?  @map("billing_account_id")
  billingAccount   BillingAccount? @relation(fields: [billingAccountId], references: [id])

  /// Mantido por retrocompat até story futura F2.5
  /// Após F2.4.5, BillingAccount.stripeCustomerId é a fonte da verdade.
  stripeCustomerId String?  @unique @map("stripe_customer_id")  // legacy

  @@index([billingAccountId])
}

model Subscription {
  // ... existing fields ...

  /// Phase 1: nullable. Phase 3: backfilled e indexed (não NOT NULL para permitir
  /// subscriptions de plataforma sem BA — futuro).
  billingAccountId String?  @map("billing_account_id")
  billingAccount   BillingAccount? @relation(fields: [billingAccountId], references: [id])

  @@index([billingAccountId])
}

model Feature {
  // ... existing fields ...

  /// Define se a quota é contada por Tenant ou por BillingAccount.
  /// Ex: MAX_PROJECTS=TENANT, MAX_TENANTS=BILLING_ACCOUNT, USERS=TENANT.
  scope FeatureScope @default(TENANT)
}

enum FeatureScope {
  TENANT
  BILLING_ACCOUNT
}
```

## Sub-stories (architect-approved breakdown)

Ordem de execução **estrita** (cada uma é um PR separado, mergeada antes da próxima começar):

### F2.4.0 — Pre-migration audit script (NEW, MUST-RUN-FIRST)
- Script standalone (`packages/database/scripts/f2-4-pre-migration-audit.ts`) que roda contra DB de prod e detecta:
  - Tenants sem nenhum user com role OWNER (data corruption — bloqueia migration)
  - Subscriptions sem tenantId válido (órfãs)
  - Stripe Customer IDs duplicados entre tenants
  - Tenants soft-deleted com Subscription ACTIVE
- Output: relatório markdown em `docs/migrations/f2-4-audit-{date}.md`. Owner roda localmente apontando pra prod e revisa antes de aprovar prosseguir.

### F2.4.1 — Schema phase 1 (backward-compat, zero breakage)
- Nova tabela `billing_accounts` (vazia).
- Coluna `tenants.billing_account_id` (nullable, FK).
- Coluna `subscriptions.billing_account_id` (nullable, FK).
- Coluna `features.scope` (default `TENANT`).
- Seed: atualiza `MAX_TENANTS` feature para `scope=BILLING_ACCOUNT`.
- Test: migration aplica + reverte limpa em DB local.

### F2.4.2 — Backfill data + Stripe migration
- Script `packages/database/scripts/f2-4-backfill.ts`:
  1. Para cada Tenant não-deletado: criar 1 BA com `ownerUserId` = primeiro User com role OWNER, `stripeCustomerId` = `tenant.stripeCustomerId` (lê via raw SQL, não pelo Prisma, porque a coluna não está no schema.base ainda).
  2. Setar `tenants.billing_account_id` e `subscriptions.billing_account_id` para o ID gerado.
  3. **Idempotente**: rodar 2× = mesmo estado (chave de detecção: `BillingAccount` cujo único Tenant tem o mesmo `stripeCustomerId`).
- Validação pós-backfill: `SELECT COUNT(*) FROM tenants WHERE deleted_at IS NULL AND billing_account_id IS NULL` deve ser 0.
- Documentar rollback: SQL pra zerar `billing_account_id` de tenants/subscriptions e dropar BAs (ordem reversa).

### F2.4.3 — Signup/checkout flow updates
- `apps/api/src/services/setup.service.ts`: novo Tenant em onboarding cria BA primeiro.
- `apps/api/src/modules/auth/services/signup.service.ts`: signup completo cria User → BA → Tenant → Subscription.
- `apps/api/src/modules/checkout/routes.ts`: prioriza `billingAccount.stripeCustomerId` > `subscription.stripeCustomerId` (legacy fallback).
- Cast `(tenant as any)?.stripeCustomerId` removido. Tipagem limpa.
- Testes: novos signups criam BA corretamente; signups legacy (pré-existente) continuam funcionando via fallback.

### F2.4.4 — Endpoint + EntitlementService generalization (MAIN VALUE)
- Novo middleware `requireBillingAccountOwner` em `apps/api/src/middleware/`.
- Novo endpoint `POST /api/billing-accounts/:id/tenants`:
  - Auth: `requireBillingAccountOwner` OR SUPER_ADMIN.
  - `requireFeature('MAX_TENANTS')` plugado.
  - Body: `{ name, slug, ... }` (mesmo shape de `POST /api/tenants` legacy).
- Novo endpoint `GET /api/billing-accounts/:id/tenants` (lista workspaces).
- Novo endpoint `GET /api/billing-accounts/me` (resolve a BA do usuário autenticado).
- `EntitlementService.resolveQuotaCurrentUsage` refactor:
  - Recebe `{ tenantId?, billingAccountId? }` em vez de só `tenantId`.
  - Despacha por `feature.scope`: `BILLING_ACCOUNT` → `prisma.tenant.count({ where: { billingAccountId, deletedAt: null } })`.
- Testes integração (Fastify.inject):
  - Plano free (MAX_TENANTS=1) com 1 tenant → 403 com body completo.
  - Plano free com 0 tenants → 201 + tenant criado.
  - Plano Pro (MAX_TENANTS=3) com 2 tenants → 201.
  - SUPER_ADMIN em plano free → 201 (bypass).
  - User não-owner da BA → 403 forbidden.
  - Cross-billing-account isolation: BA-A com 3 tenants não conta na contagem de BA-B.

### F2.4.5 — Constraints + cleanup (Phase 3)
- ALTER TABLE: `tenants.billing_account_id NOT NULL`.
- Update docs: `apps/docs/content/platform/features/PLANS_PRODUCTS.mdx` explica multi-workspace.
- Update F2.1 + F2.3 stories com link "MAX_TENANTS agora ativo via F2.4".
- Endpoint `POST /api/tenants` (SUPER_ADMIN-only) recebe banner em swagger: "internal-platform-only, prefer /api/billing-accounts/:id/tenants".
- ADR em `docs/adr/2026-04-18-billing-account-vs-extending-user.md`.

### F2.4.6 (DEFERRED — não bloqueia ativação MAX_TENANTS)
- `BillingAccountMember` table (multi-owner, role OWNER/ADMIN/VIEWER).
- Endpoint de transferência de ownership.
- Auditoria de mudanças de billing.
- Será criada como story F2.4.6 separada quando o produto pedir.

## Riscos + Mitigações

| Risco | Mitigação |
|---|---|
| Backfill lock no `tenants` em prod | F2.4.1 separa schema change (rápido) de F2.4.2 backfill (longo). Backfill rodado fora da janela de schema. |
| Stripe Customer legacy desalinhado | F2.4.0 audit detecta duplicates antes de migrar. F2.4.2 reconcilia via raw SQL. |
| Tenant sem OWNER user | F2.4.0 audit **bloqueia** prosseguir até owner corrigir manualmente. |
| Breaking change pra clientes do `POST /api/tenants` | Endpoint preservado. Novo endpoint coexiste. Deprecation só em story futura. |
| Race condition entre signup parallel (2 tenants criados antes do count atualizar) | `EntitlementService.validateAction` usa transação isolada. Documentar limitação conhecida (1ª chamada vence). |
| `Tenant.stripeCustomerId` sumir antes do backfill ler | F2.4 estritamente sequencial. F2.4.5 NÃO remove o campo (deferido para F2.5). |
