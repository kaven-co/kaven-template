# F2.4 — Multi-Workspace Ownership Model

> **Epic:** 4 (Framework — Pricing Enforcement) | **Sprint:** F2 | **Agent:** @pm (story) → @architect (design) → @dev (impl) → @qa
> **Priority:** P1 (ativa a quota MAX_TENANTS de fato)
> **Depends on:** F2.1, F2.2, F2.3 (merged em PR #113 e #114 em 2026-04-18)
> **Blocker para:** activação real de `MAX_TENANTS` na API

---

## Contexto

O sprint F2 entregou infra completa de enforcement de quota por plano (F2.1 MAX_TENANTS + F2.2 MAX_PROJECTS + F2.3 hardening). F2.2 está **ativo em produção** via `requireFeature('MAX_PROJECTS')` em `POST /api/v1/projects`. F2.1 **não está ativo** porque:

1. O único endpoint de criação de tenant (`POST /api/tenants`) é **SUPER_ADMIN-only**, e SUPER_ADMIN bypassa quotas por design (feature-guard.middleware.ts).
2. O modelo de dados atual é **1 Tenant → 1 Subscription** (relacionamento 1:1 via `Subscription.tenantId` com constraint unique). Não existe um conceito de "billing account" que possa owner N tenants.
3. Com o modelo atual, `MAX_TENANTS=3` por plano Pro não tem como ser interpretado — o plano está na Subscription, que já pertence a um único Tenant.

A infra de F2.1 (seed de features + `EntitlementService.resolveQuotaCurrentUsage` contando tenants ativos globalmente) **permanece pronta** para ser plugada no endpoint certo assim que o modelo de ownership existir.

Esta story resolve isso desenhando e implementando o modelo de multi-workspace ownership.

---

## User Story

**Como** um customer Pro do Kaven,
**Quero** poder criar e gerenciar até 3 workspaces (tenants) distintos sob uma única assinatura,
**Para que** eu possa separar contextos (ex: agências com múltiplos clientes, grupos empresariais com filiais) sem pagar N assinaturas.

**Como** engenheiro de plataforma,
**Quero** um modelo canônico de "BillingAccount owner-of Tenants" com enforcement automático,
**Para que** o campo `MAX_TENANTS` dos planos deixe de ser vaporware e passe a regular criação de workspaces de verdade.

## Design de Arquitetura (draft — requer validação @architect)

### Opção A (recomendada): Introduzir `BillingAccount` (entity nova)

```
BillingAccount                     Tenant
─────────────                      ──────
id                                 id
stripeCustomerId  (migrado)        name, slug, status, …
subscriptionId (1:1 com Subsc.) ←  billingAccountId (FK novo, NOT NULL)
ownerUserId (dono/billing admin)   ─────
createdAt / updatedAt
```

**Migração em 3 fases:**

1. **Fase 1 — Schema (backward-compatible):**
   - Criar tabela `billing_accounts` vazia.
   - Adicionar coluna `billing_account_id` nullable em `tenants`.
   - Adicionar coluna `billing_account_id` em `subscriptions` (nullable).

2. **Fase 2 — Backfill:**
   - Para cada Tenant existente: criar 1 BillingAccount (1:1) com `stripeCustomerId` herdado do Tenant, `ownerUserId` = primeiro user com role OWNER do tenant.
   - Ligar Subscription existente à BillingAccount.
   - Mover `Tenant.stripeCustomerId` → `BillingAccount.stripeCustomerId` (TD-15).
   - Validar que toda Subscription tem exatamente 1 BillingAccount e todo Tenant tem exatamente 1 BillingAccount.

3. **Fase 3 — Constraints + endpoints:**
   - Tornar `Tenant.billingAccountId` NOT NULL.
   - Mover relationship Subscription ↔ Tenant para Subscription ↔ BillingAccount.
   - Novo endpoint `POST /api/billing-accounts/:id/tenants` (cria novo workspace sob billing account existente).
   - Plugar `requireFeature('MAX_TENANTS')` neste endpoint — contagem: tenants ativos desta BillingAccount (não globais).
   - `resolveQuotaCurrentUsage` passa a receber `billingAccountId` em vez de `tenantId` para quota MAX_TENANTS.
   - Deprecar `POST /api/tenants` (SUPER_ADMIN-only) ou mantê-lo para operações de plataforma.

### Opção B (alternativa — rejected pending analyst review): estender `User` como billing holder

- Pros: menos schema change (User já tem relação com Tenant)
- Cons: acopla billing identity ao usuário pessoal (problemas quando usuário sai da empresa, quando há múltiplos billing admins, quando a empresa é a entidade fiscal)
- Decisão provisória: **não seguir** — BillingAccount como entidade dedicada é o padrão SaaS canônico (Stripe Customer ≈ BillingAccount, não User).

## Acceptance Criteria

### AC1 — Schema migrado com backfill seguro
- [ ] Migration Prisma em 3 fases (criar tabela → backfill → constraints) versionada e aplicável em produção sem downtime.
- [ ] Backfill idempotente (rodar 2× = mesmo estado).
- [ ] Script de validação pós-backfill: `SELECT COUNT(*) FROM tenants WHERE billing_account_id IS NULL` = 0 após fase 3.
- [ ] Rollback documented: como reverter fase 3 → fase 2 → fase 1.

### AC2 — Endpoints funcionais
- [ ] `POST /api/billing-accounts/:id/tenants` cria tenant sob billing account existente, com `requireFeature('MAX_TENANTS')` ativo.
- [ ] 403 ao estourar quota com body `{ featureCode: 'MAX_TENANTS', currentUsage, limit, upgradeUrl }`.
- [ ] SUPER_ADMIN bypassa.
- [ ] Endpoint antigo `POST /api/tenants` (SUPER_ADMIN-only) documentado como "uso interno de plataforma" ou deprecado.

### AC3 — `EntitlementService` atualizado
- [ ] `resolveQuotaCurrentUsage(billingAccountId, 'MAX_TENANTS')` conta tenants ativos **desta billing account** (não globais).
- [ ] Tenants com `deletedAt IS NOT NULL` excluídos da contagem.
- [ ] Teste unitário cobrindo: 0 tenants, N tenants, N+deletados, cross-billing-account isolation.

### AC4 — Migration de dados (TD-15 Stripe Customer)
- [ ] `Tenant.stripeCustomerId` movido para `BillingAccount.stripeCustomerId`.
- [ ] Código em `apps/api/src/modules/checkout/routes.ts` ajustado: prioridade passa a ser `billingAccount.stripeCustomerId` > `subscription.stripeCustomerId` (legacy).
- [ ] Cast `(tenant as any)?.stripeCustomerId` removido (tipagem limpa).
- [ ] Backfill copia `tenant.stripeCustomerId` para `billingAccount.stripeCustomerId` antes de remover a coluna do Tenant.

### AC5 — Testes
- [ ] Teste integração Fastify.inject para `POST /api/billing-accounts/:id/tenants`:
   - Plano free (MAX_TENANTS=1) com 1 tenant → 403
   - Plano free com 0 tenants → 201 + tenant criado
   - Plano Pro (MAX_TENANTS=3) com 2 tenants → 201
   - SUPER_ADMIN em plano free → 201 (bypass)
- [ ] Teste unitário `resolveQuotaCurrentUsage` cobrindo isolamento cross-billing-account.

### AC6 — Docs
- [ ] ADR em `docs/adr/` documentando decisão "BillingAccount vs extending User".
- [ ] Atualizar `apps/docs/content/platform/features/PLANS_PRODUCTS.mdx` explicando multi-workspace.
- [ ] Atualizar `F2.1` e `F2.3` com link para F2.4 (DONE: MAX_TENANTS agora ativo).

## Riscos

| Risco | Mitigação |
|---|---|
| Backfill lock no `tenants` durante produção | Migration em 3 fases com backfill idempotente fora da janela de schema change |
| Stripe Customer legacy desalinhado | Script de reconciliação: para cada Subscription, comparar `stripeCustomerId` entre BillingAccount e Stripe API |
| Usuário tem tenants em múltiplos planos hoje | NÃO DEVERIA existir (1 tenant = 1 subscription hoje). Script de auditoria pré-migration para detectar anomalias |
| Breaking change pra quem usa `POST /api/tenants` | Manter endpoint + deprecation notice. Novos clientes usam `/api/billing-accounts/:id/tenants` |
