# F2.4.5 — Drop legacy Stripe columns + remove ALLOW_LEGACY_TENANT_STRIPE_ID flag

**Status:** READY
**Sprint:** F2
**Epic:** EPIC-004 Framework
**Parent:** F2.4 Multi-Workspace Ownership
**Depends on:** F2.4.4a, F2.4.4b, F2.4.4c
**Blocks:** —
**Effort:** 2–3h

---

## Contexto

F2.4 introduziu `BillingAccount.stripeCustomerId` como fonte autoritativa (ADR-0001).
Desde F2.4.3.1, novos tenants **nunca** populam `Tenant.stripeCustomerId`.
Desde F2.4.4b, novos webhooks **nunca** escrevem `Subscription.stripeCustomerId`.

Esta story fecha o ciclo: remove as colunas legadas e a feature flag de transição,
tornando `BillingAccount` a única fonte de verdade sem fallbacks no código.

### Gate de produção (pré-requisito humano)

Antes de executar qualquer migration nesta story:

1. `ALLOW_LEGACY_TENANT_STRIPE_ID=false` em prod **por ≥ 7 dias** sem incidentes.
2. Rodar os invariant checks SQL do ADR-0001 (ver abaixo) — ambos devem retornar 0 rows.

Esta story pode ser implementada e testada em staging/development sem o gate,
mas o `pnpm prisma migrate deploy` em prod deve aguardar os 7 dias.

---

## Invariant checks SQL (rodar antes do deploy em prod)

```sql
-- 1. Nenhum Tenant com BA deve ter stripe_customer_id populado pós-F2.4.3.1.
-- Esperado: 0 rows. Qualquer row = tenant legacy sem backfill ou bug de signup.
SELECT t.id, t.stripe_customer_id
FROM "Tenant" t
JOIN billing_accounts ba ON ba.id = t.billing_account_id
WHERE t.stripe_customer_id IS NOT NULL
  AND t.created_at > '<deploy_f2_4_3_1_timestamp>';

-- 2. Nenhuma Subscription criada pós-F2.4.4b deve ter stripe_customer_id populado.
-- Esperado: 0 rows. Qualquer row = webhook ainda escrevendo campo legado.
SELECT COUNT(*) AS violations
FROM subscriptions
WHERE stripe_customer_id IS NOT NULL
  AND created_at > '<deploy_f2_4_4b_timestamp>';
```

## Escopo

### 1. Migration — drop `Tenant.stripeCustomerId`

- Criar migration Prisma: `ALTER TABLE "Tenant" DROP COLUMN stripe_customer_id`.
- Remover campo `stripeCustomerId` do model `Tenant` em `schema.prisma`.
- Nome sugerido: `20260420000000_f2_4_5_drop_tenant_stripe_customer_id`.

### 2. Migration — drop `Subscription.stripeCustomerId`

- Criar migration Prisma: `ALTER TABLE subscriptions DROP COLUMN stripe_customer_id`.
- Remover campo `stripeCustomerId` do model `Subscription` em `schema.prisma`.
- Nome sugerido: `20260420000001_f2_4_5_drop_subscription_stripe_customer_id`.

### 3. Remover feature flag `ALLOW_LEGACY_TENANT_STRIPE_ID`

- `apps/api/src/config/env.ts`: remover campo `ALLOW_LEGACY_TENANT_STRIPE_ID` do `envSchema` e `isLegacyTenantStripeIdAllowed()`.
- `apps/api/src/modules/checkout/routes.ts`: remover bloco `if (env.ALLOW_LEGACY_TENANT_STRIPE_ID)` — manter apenas a leitura de `BillingAccount.stripeCustomerId`.
- `apps/api/src/modules/billing/services/payment.service.ts`: remover `resolveStripeCustomerId` fallback para `Tenant.stripeCustomerId`; leitura exclusiva de `billingAccount.stripeCustomerId`.
- Qualquer outro call-site que leia `stripeCustomerId` de `Tenant` ou `Subscription` — grep e migrar.

### 4. Grep de call-sites legados

Antes de fechar o PR, rodar:

```bash
grep -rn "stripeCustomerId\|stripe_customer_id" apps/api/src/ \
  --include="*.ts" | grep -v "billingAccount\|BillingAccount\|spec\|test\|migration"
```

Esperado: 0 results. Qualquer resultado = call-site não migrado → bloqueia o merge.

### 5. Testes

- Atualizar mocks/specs que referenciem `Tenant.stripeCustomerId` ou `Subscription.stripeCustomerId`.
- Adicionar test para `checkout/routes.ts` confirmando que fallback legacy não existe mais.
- Adicionar test para `payment.service.ts` confirmando leitura exclusiva de BA.
