## Escopo

### 1. Secure logger — completar cobertura PII

1.1. Adicionar regex de email para **value-level** redaction em `sanitizeValue`. Emails em string interpolada ou sob keys não-sensitive (ex: `{ to: "x@y.com" }`, `{ message: "User x@y.com failed" }`) passam hoje.

1.2. Expandir `SENSITIVE_KEYS` com: `to`, `from`, `recipient`, `cc`, `bcc`, `replyto`, `reply_to`. Uso documentado: nenhum log legítimo precisa do valor cru desses campos.

1.3. `maskStripeId` atual usa `indexOf('_')`, vazando suffix `test` para IDs como `cs_test_xxx`. Trocar por fatiamento de length fixa (`slice(0, 4)`) — não revela env.

1.4. Adicionar 5 casos adversariais ao `secure-logger.spec.ts`: circular ref, prototype pollution, Symbol keys, `Buffer`/`Map`/`Set` com PII, `Date`/`Error` aninhado.

### 2. Stripe orphan sweep — safe para prod

2.1. **Age-gate (HIGH latente):** skip Customers criados há < 3600s. Stripe search tem eventual consistency ~1min; sem o gate, sweep pode deletar Customer recém-criado cuja tx signup ainda não commitou.

2.2. **Idempotent delete:** catch `Stripe.errors.StripeInvalidRequestError` com `code === 'resource_missing'` (404) como no-op em `skipped`, não `errors`. Re-runs pós-falha parcial não explodem mais.

2.3. **Audit sidecar JSON:** antes de `exit`, serializar `report` completo em `./logs/stripe-sweep-<ISO-timestamp>.json` com `{dryRun, confirmed, startedAt, finishedAt, deleted, orphans, dangling, errors, skipped, operator: process.env.USER}`. File com `mode: 0o600`. Ignorar em `.gitignore`.

2.4. **Live-mode sanity log:** no boot, logar prefix do `STRIPE_SECRET_KEY` (mascarado: `sk_live_***` vs `sk_test_***`). Se `sk_live` + `--delete`, exigir flag adicional `ALLOW_STRIPE_LIVE_DELETE=yes` para não fatalizar por acidente.

### 3. Backfill script — cobrir caso da BA stale por completo

3.1. **Subscription re-link:** no path da guarda H (tenant com BA stale), o `WHERE billing_account_id IS NULL` nas subscriptions deixa subs pré-ligadas à BA velha órfãs de BA viva. Expandir predicate para `WHERE billing_account_id IS NULL OR billing_account_id = <staleBaId>`. Post-validate ganha check: `JOIN billing_accounts ba ON ba.id = s.billing_account_id WHERE ba.deleted_at IS NOT NULL` = 0.

3.2. **Race counter reclassification:** race legítima (signup venceu) vira `tenantsSkippedByRace`, não `errors`. `process.exit(1)` só dispara em erros reais.

### 4. ADR-0001 — drift cleanup

4.1. **§3 Webhook handlers**: marcar explicitamente `webhook deve retry (HTTP 202)` como `[TARGET: F2.4.4b — não implementado hoje]`. Webhook atual sempre retorna 200 intencionalmente.

4.2. **§4 Payment service**: renomear cabeçalho para `Payment service (subscription.service, invoice.service — migração pendente em F2.4.4c)`. Adicionar linha: "Hoje leem via `Subscription`/`Invoice` + `tenantId`; migração para `billingAccountId` em F2.4.4c."

4.3. **§7 Admin/Support readers (novo)**: painéis admin/support hoje podem ler `Tenant.stripeCustomerId` como fallback; novos readers devem consumir `tenant.billingAccount.stripeCustomerId`; gate de leitura legacy atrás de feature flag `ALLOW_LEGACY_TENANT_STRIPE_ID` (F2.4.4a implementa a flag).

4.4. **Plano de retirada F2.4.5 — step 0**: "antes do DROP COLUMN, verificar invariante `Subscription.stripeCustomerId IS NULL` para todas subs criadas pós-F2.4.4 deploy. Falhar a migração se encontrar row populada — significa webhook não foi migrado corretamente em F2.4.4b."

4.5. Atualizar story `F2.4.3.2-micro-hardening.md` AC5: "5 subseções" → "6 subseções" (ADR entregou 6, stave delivered ≥ planned).

### 5. F2.4.4 story split
