# F2.2 — MAX_PROJECTS Plan Enforcement

> **Epic:** 4 (Framework — Pricing Enforcement) | **Sprint:** F2 | **Agent:** @dev
> **Priority:** P0
> **Depends on:** F2.1 (mesmo padrão de PlanFeature)

---

## Contexto

O kaven-framework **não tem** nenhum enforcement de limite de projetos por tier. A licença é por usuário, não por projeto. O campo `Installation.projectId` existe mas é apenas rastreio — sem nenhuma lógica de quota. Não existe `MAX_PROJECTS`, `projectLimit` ou campo similar no schema ou no seed.

Esta story constrói o enforcement **do zero**, seguindo exatamente o mesmo padrão de F2.1 (`MAX_TENANTS`) usando a infra de `PlanFeature.limitValue` + `requireFeature()` já existente.

**Limites aprovados (decisão final 2026-04-17):**
- Starter: **1 projeto**
- Builder: **3 projetos**
- Pro: **10 projetos**

*Nota: "projeto" no contexto do framework = uma instalação/deployment independente do kaven. Starter é mono-projeto por design — foco e simplicidade.*

---

## User Story

**Como** usuário tentando criar um novo projeto além do limite do meu plano,
**Quero** receber uma resposta clara de bloqueio com informação de upgrade,
**Para que** eu entenda o limite do meu plano e tenha um caminho direto para expandir.

### Feature Creation (Seed)
- [ ] `MAX_PROJECTS` adicionado em `capabilities.seed.ts` com `type: QUOTA`
- [ ] `PlanFeature` seed registra os limites: `{ STARTER: 1, BUILDER: 3, PRO: 10 }`
- [ ] Seed é idempotente (pode ser re-executado sem duplicatas)

### Identificação do Endpoint
- [ ] Identificar endpoint de criação de projeto/installation (ex.: `POST /installations` ou equivalente)
- [ ] Documentar path exato no PR como evidência

### Enforcement no Endpoint
- [ ] `requireFeature('MAX_PROJECTS')` adicionado no endpoint identificado
- [ ] Quando project count >= limite do plano, retorna 403 com body:
  ```json
  {
    "error": "Feature not available",
    "featureCode": "MAX_PROJECTS",
    "currentUsage": 1,
    "limit": 1,
    "availableUpgrades": [...],
    "upgradeUrl": "/pricing"
  }
  ```
- [ ] Admin bypass funcionando (admins ignoram o check)
- [ ] Counter de projetos conta corretamente para o usuário/tenant correto

### Testes
- [ ] Teste: Starter cria 1 projeto → OK; tenta criar 2º → 403 com `featureCode: MAX_PROJECTS`
- [ ] Teste: Builder cria 3 projetos → OK; tenta criar 4º → 403
- [ ] Teste: Pro cria até 10 projetos → OK; tenta criar 11º → 403
- [ ] Teste: Admin ignora o check
- [ ] Todos os testes existentes passando (sem regressões)

## Technical Notes

**Arquivos de referência para o padrão existente:**
- `packages/database/prisma/seeds/capabilities.seed.ts` — seguir padrão de F2.1 (`MAX_TENANTS`)
- `apps/api/src/middleware/feature-guard.middleware.ts` — `requireFeature()` QUOTA
- `packages/database/prisma/schema.prisma` — verificar model `Installation` e relações

**Investigação necessária antes de implementar:**
1. Verificar se existe model `Project` separado de `Installation`
2. Localizar o endpoint de criação de projeto: buscar `POST` em `apps/api/src/modules/`
3. Entender como o counter de projetos por usuário é calculado (query necessária)

**Pattern a seguir (mesmo de F2.1):**
```typescript
// Em capabilities.seed.ts
{
  code: 'MAX_PROJECTS',
  name: 'Maximum Projects',
  description: 'Maximum number of projects per license',
  type: 'QUOTA',
  category: 'LIMITS',
  planFeatures: {
    create: [
      { plan: { connect: { name: 'STARTER' } }, limitValue: 1 },
      { plan: { connect: { name: 'BUILDER' } }, limitValue: 3 },
      { plan: { connect: { name: 'PRO' } }, limitValue: 10 },
    ]
  }
}
```

**Counter query** — exemplo baseado no padrão existente:
```typescript
// Contar projetos ativos do usuário antes de criar novo
const projectCount = await prisma.installation.count({
  where: { userId: request.user.id, status: 'ACTIVE' }
});
// requireFeature vai comparar com limitValue do plano
```
