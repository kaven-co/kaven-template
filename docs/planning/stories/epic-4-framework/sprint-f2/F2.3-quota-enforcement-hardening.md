## Contexto

Relatório Quinn 2026-04-18 fechou **F2.2 como PASS** e **F2.1 como CONCERNS**. A infraestrutura (seed + `EntitlementService.resolveQuotaCurrentUsage` + middleware) está correta e testada (211/211 unit tests passando). Mas sobraram 6 pendências objetivas:

| Ref | Concern | Criticidade |
|-----|---------|-------------|
| C1 | `requireFeature('MAX_TENANTS')` plugado em `POST /api/tenants` que é SUPER_ADMIN-only, e SUPER_ADMIN bypassa o middleware → **enforcement nunca dispara em runtime**. | 🔴 alta |
| C2 | AC das stories F2.1/F2.2 mandava modificar `capabilities.seed.ts`, dev (corretamente) modificou `seed.ts`/`feature-flags-projects.ts`. Divergência não documentada em Dev Notes. | 🟡 baixa |
| C3 | "Admin bypass" interpretado como SUPER_ADMIN (não TENANT_ADMIN). Razoável, mas sem registro formal de decisão. | 🟡 baixa |
| C4 | Idempotência do seed garantida por construção (`upsert`) mas **não validada empiricamente** (`prisma db seed` 2×). | 🟡 média |
| C5 | Testes atuais batem direto no `EntitlementService`. Falta **teste de integração Fastify.inject** cobrindo rota→middleware→service para `POST /api/v1/projects`. | 🟡 média |
| C6 | `apps/api/package.json` não define script `lint` → AC de DoD "`npm run lint` passando" é vacuosamente satisfeito. | 🟢 baixa |

**Decisão de produto sobre C1 (tomada em 2026-04-18, orquestrador Orion + owner):**
A feature real de "MAX_TENANTS = 3/10/-1 por plano" pressupõe um conceito de **ownership Tenant↔Billing** que **ainda não existe** no schema (hoje cada tenant tem sua própria subscription 1-para-1). A infra de quota (seed + service) fica **parada aguardando** a feature de multi-workspace, que será desenhada em **F2.4 — Multi-Workspace Ownership Model** (story de produto + arquitetura, não incluída neste sprint).

Portanto, **esta story remove o código morto** (`requireFeature('MAX_TENANTS')` do route SUPER_ADMIN-only) e preserva a infra para quando F2.4 entrar.

## Technical Notes

### Ponto de partida (código atual pós-F2.1/F2.2)

```ts
// apps/api/src/modules/tenants/routes/tenant.routes.ts
fastify.post('/', {
  preHandler: [
    authMiddleware,
    requireSuperAdmin,
    requireFeature('MAX_TENANTS'), // ← REMOVER: dead code (SUPER_ADMIN bypassa)
  ],
  handler: tenantController.create.bind(tenantController),
});
```

Depois da remoção, fica:

```ts
fastify.post('/', {
  // MAX_TENANTS quota infra (seed + EntitlementService) está pronta mas
  // aguarda F2.4 (Multi-Workspace Ownership Model) para ser plugada no
  // endpoint correto. Rota atual é SUPER_ADMIN-only e bypassa quotas por design.
  preHandler: [authMiddleware, requireSuperAdmin],
  handler: tenantController.create.bind(tenantController),
});
```

### Dev Notes

**Decisão 1 — Seed location (C2):**
A story F2.1 mandava modificar `packages/database/prisma/seeds/capabilities.seed.ts`. Esse arquivo **não** é o canal real do monorepo para features de billing. Features de plano entram em:
- `packages/database/prisma/seed.ts` (seed principal, features gerais)
- `packages/database/prisma/seeds/feature-flags-{module}.ts` (features por módulo)

Dev seguiu o padrão real. Story F2.1 estava desatualizada.

**Decisão 2 — Admin bypass = SUPER_ADMIN (C3):**
- TENANT_ADMIN deve respeitar seu próprio plano (se não, billing não funciona).
- SUPER_ADMIN = operador de plataforma, precisa criar tenants/projetos pra QA, migração, debug, etc.
- Bypass aplicado em `feature-guard.middleware.ts:24-26` (`if (user?.role === 'SUPER_ADMIN') return`).

**Decisão 3 — MAX_TENANTS é dead code hoje (C1):**
Relatório Quinn mostrou que `POST /api/tenants` é SUPER_ADMIN-only e SUPER_ADMIN bypassa quotas. Logo, `requireFeature('MAX_TENANTS')` nunca dispara. Além disso, o modelo de dados atual não suporta "billing account owns N tenants" (cada tenant tem sua subscription 1-para-1). A feature real entra em F2.4.
