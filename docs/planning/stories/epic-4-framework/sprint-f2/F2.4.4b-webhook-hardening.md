# F2.4.4b — Webhook hardening: BA-resolution + transient retry (Stripe) + stop writing Subscription.stripeCustomerId

**Status:** READY
**Sprint:** F2
**Epic:** EPIC-004 Framework
**Parent:** F2.4 Multi-Workspace Ownership
**Depends on:** F2.4.4a (AuthZ helpers + flag)
**Blocks:** F2.4.5 (constraints + drop column)
**Effort:** 3–4h

---

## Contexto

Webhook handler atual (pré-F2.4.4b):
- Escreve `Subscription.stripeCustomerId` sempre (campo marcado como legado em ADR-0001 §3).
- Sempre retorna HTTP 200 mesmo se o BA do tenant não foi resolvido ainda (race com signup commit ou webhook antes de signup).

Ambos bloqueiam F2.4.5 (drop de `Subscription.stripeCustomerId` falharia invariant check do ADR).

---

## Escopo

### 1. Resolver `billingAccountId` em events que criam/atualizam Subscription

- Evento `customer.subscription.created` / `.updated` / `.deleted`: recuperar `tenantId` via `event.data.object.metadata.tenantId` (signup já seta). Resolver BA via `tenant.billingAccountId`.

### 2. Stop writing `Subscription.stripeCustomerId`

- Escrita em `Subscription` passa a popular **apenas** `billingAccountId`. Campo legado permanece NULL para novas rows.

### 3. BA-not-found-yet vs legado (sem confundir com retry do Stripe)

**Fato Stripe:** webhooks são re-entregues quando o endpoint responde **fora da faixa 2xx** (timeout ou erro). Qualquer **2xx incl. 202** conta como **sucesso** — o Stripe **não** re-tenta por causa de 202.

**Árvore de decisão:**

1. **Tenant não existe** (hard-deleted / ID inválido): **HTTP 200** + log `warn` — evento órfão; não há o que materializar.
2. **Tenant existe, `billingAccountId` NULL, legado pré-F2.4** (heurística: tenant criado antes do cutoff F2.4.1 **ou** flag `ALLOW_LEGACY_TENANT_STRIPE_ID` + ausência de BA esperada): seguir ADR §3 — materializar `Subscription` com `billingAccountId = NULL` + warning. **HTTP 200**.
3. **Tenant existe, `billingAccountId` NULL, caso transitório** (ex.: signup tx ainda não commitou — heurística documentada em implementação, ex. tenant muito recente + metadata de signup): **uma** destas estratégias (escolher na implementação, documentar no handler):
   - **A — Preferida:** **HTTP 200** + enfileirar evento em job interno idempotente que re-tenta até BA aparecer (desacopla do schedule de retry do Stripe).
   - **B:** **HTTP 503** (ou 500) para acionar **retry automático do Stripe** — *somente* se houver garantia de **não** loop infinito em tenants legados permanentemente sem BA (nunca retornar 503 para o caso 2).

- Limite: se usar estratégia B, após N falhas Stripe desiste — logar `error` e considerar alerta operacional.

### 4. Testes

- Test: event com `tenant.billingAccountId` presente → writes `Subscription.billingAccountId`, NOT `stripeCustomerId`.
- Test: event com `tenant.billingAccountId === null` transitório → HTTP 200 + job enqueued **ou** HTTP 503 (conforme estratégia escolhida), nunca 202 como “retry signal”.
- Test: event com tenant não-existente → HTTP 200 + error log.
- Test: invariant — `Subscription.stripeCustomerId IS NULL` para todas subs criadas pós-F2.4.4b.

## Acceptance Criteria

### AC1 — BA resolution
- [ ] Handler extrai `tenantId` do metadata e resolve BA via tenant.
- [ ] Subscription upsert escreve `billingAccountId`, nunca `stripeCustomerId`.

### AC2 — Transient BA + Stripe semantics
- [ ] BA-not-found-yet **transitório** → **HTTP 200 + fila interna** *ou* **HTTP 503** (estratégia A ou B da §3), **nunca** HTTP 202 como substituto de retry.
- [ ] Tenant inexistente → HTTP 200.
- [ ] Legado `billingAccountId` NULL permanente → HTTP 200 + materialização ADR §3 (sem 503 em loop).
- [ ] Warning log inclui `stripeSubscriptionId` mascarado.

### AC3 — Invariant
- [ ] Post-deploy check: `SELECT count(*) FROM subscriptions WHERE stripe_customer_id IS NOT NULL AND created_at > '<deploy_f2_4_4b>'` retorna 0.

### AC4 — Quality gates
- [ ] Tests + typecheck + lint verdes.

### AC5 — QA gate
- [ ] security-sentinel + data-integrity-guardian review pass.
