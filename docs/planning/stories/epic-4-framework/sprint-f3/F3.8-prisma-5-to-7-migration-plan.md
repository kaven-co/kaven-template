## Contexto

Auditoria técnica de 2026-04-20 identificou que o ecossistema está rodando Prisma `^5.22.0` em três pontos:

- `package.json` (root)
- `apps/api/package.json`
- `packages/database/package.json`

Prisma 6.x e 7.x já foram lançados com breaking changes na client API e no CLI. Permanecer no 5.x expõe o projeto a:
- Falta de suporte para novos tipos do PostgreSQL (e.g., tipos Neon-specific, JSON schema validation).
- Breaking changes acumulados que ficam cada vez mais custosos de resolver com o tempo.
- Incompatibilidades com versões futuras do Node.js e TypeScript.

Esta story cobre o **plano de migração** — análise de breaking changes, mapeamento de call-sites afetados e uma implementação incremental segura. A migration deve ser feita em branch isolada com CI verde como gate obrigatório.

---

## Escopo

### 1. Análise de breaking changes Prisma 5 → 6

- Consultar o changelog oficial do Prisma 6.x para identificar breaking changes.
- Mapear impacto em:
  - `PrismaClient` instantiation (mudanças em `datasourceUrl`, `log`, `errorFormat`).
  - `$transaction` API (mudanças em isolation level, timeout, max wait).
  - `$extends` e middleware API.
  - Tipos gerados (`Prisma.XxxGetPayload`, `Prisma.XxxWhereInput`, etc.) — verificar se houve renaming.
  - CLI commands (`prisma migrate`, `prisma generate`, `prisma db push`).
  - `prisma-client-js` generator options.
- Documentar breaking changes relevantes em comentário da PR.

### 2. Análise de breaking changes Prisma 6 → 7

- Repetir o processo para 6 → 7.
- Identificar se a migration incremental (5→6→7) é preferível à direta (5→7).
- Recomendar estratégia no PR.

### 3. Mapeamento de call-sites afetados

Executar busca nos arquivos `apps/api/src/` e `packages/database/` para identificar padrões que mudam:

```bash
# Exemplos de patterns a verificar:
grep -rn "PrismaClient" apps/api/src/ packages/database/
grep -rn "\$transaction" apps/api/src/
grep -rn "\$extends" apps/api/src/
grep -rn "Prisma\." apps/api/src/ | grep -v "import"
```

- Gerar lista de arquivos que precisarão de mudança.
- Estimar esforço por arquivo.

### 4. Implementação da migration

- Criar branch `chore/prisma-7-upgrade`.
- Atualizar `prisma` e `@prisma/client` nos três `package.json` para `^7.x.x` (latest stable).
- Executar `pnpm install` e resolver erros de type.
- Executar `pnpm prisma generate` e corrigir tipos quebrados.
- Corrigir call-sites identificados no passo 3.
- Garantir que `pnpm prisma migrate deploy` funciona sem erros.

### 5. Validação

- `pnpm test --filter api` — suite completa deve passar (sem novos failures).
- `pnpm typecheck` verde em todos os workspaces.
- `pnpm prisma validate` verde.
- `pnpm prisma generate` sem warnings de deprecation.

## Acceptance Criteria

### AC1 — Análise documentada
- [ ] Breaking changes Prisma 5→6 e 6→7 listados no PR description ou em `docs/architecture/`.
- [ ] Lista de call-sites afetados com arquivo e linha.
- [ ] Estratégia de migration (incremental vs direta) documentada com justificativa.

### AC2 — Versão unificada
- [ ] `prisma` e `@prisma/client` em versão `^7.x.x` nos três `package.json` (`root`, `apps/api`, `packages/database`).
- [ ] Nenhuma ocorrência de `^5.` ou `^6.` nas dependências Prisma.

### AC3 — Sem regressões
- [ ] `pnpm test --filter api` passa sem novos failures (baseline: ~2100 tests).
- [ ] `pnpm typecheck` verde em todos os workspaces afetados.
- [ ] `pnpm prisma validate` verde.
- [ ] `pnpm prisma generate` sem warnings de deprecation sobre APIs removidas.

### AC4 — Migrations em produção
- [ ] `prisma migrate deploy` executa sem erros contra banco Neon de staging.
- [ ] Zero drift detectado (checksum das migrations históricas intacto).

## Out of scope

- Upgrade de outras dependências não relacionadas ao Prisma.
- Migração de outros repos do ecossistema (marketplace, template) — stories separadas.
- Mudança de schema do banco ou criação de novas migrations.
- Adoção de features novas do Prisma 7 além do necessário para migração.
