## 1. Contexto e Justificativa Técnica (Deep Analysis)

O ecossistema Kaven encontra-se em um estado de **fragmentação técnica**. O **Marketplace** (provedor de módulos) migrou para Prisma 7, enquanto o **Framework** (este repo) permanece na v5.22.0.

### 🔴 Riscos da Permanência na v5:
- **Incompatibilidade de Tipos:** O pacote `@kaven/shared` gerado na v5 não é compatível com os esquemas v7 do Marketplace, causando erros de `Type mismatch` em operações cross-module.
- **Overhead de Conexão (TCP):** O driver padrão `postgresql` sofre de latência excessiva (vimos >300ms nos testes remotos) devido ao handshake TCP em cada nova instância no Cloud Run.
- **Engines Obsoletas:** O Prisma 7 removeu suporte a engines binárias legadas, movendo para Wasm, o que é mandatório para otimização de cold-start em Serverless.

### 🟢 Benefícios da Migração v7 + Neon Adapter:
- **Neon WebSocket Driver:** Redução drástica da latência através do `@prisma/adapter-neon`, permitindo que os testes de integração passem dentro do SLA de 100ms.
- **Relation Load Strategy:** Uso da nova API `relationLoadStrategy: 'join'` para otimizar queries complexas (evitando o problema de N+1 nativo do Prisma).

## 3. Escopo Técnico Detalhado

### 3.1. Arquitetura do Schema (Fase 1)
- Atualizar `generator client` no `packages/database/prisma/schema.prisma`:
  - Ativar `previewFeatures = ["driverAdapters"]` (se necessário para a versão exata).
  - Configurar `engineType = "library"` (ou conferir padrão Wasm da v7).
  - Definir `binaryTargets` explicitamente para `["native", "linux-musl-openssl-3.0.x"]` para garantir compatibilidade com o runner do GitHub e Cloud Run.

### 3.2. Unificação de Dependências (Fase 2)
- O `@dev` deve realizar o upgrade sincronizado:
  - `root`: `prisma@7.x`, `@prisma/client@7.x`
  - `apps/api`: `@prisma/client@7.x`, `@prisma/adapter-neon`, `@neondatabase/serverless`, `ws`, `@types/ws`.
  - `packages/database`: `prisma@7.x`, `@prisma/client@7.x`.
- **Atenção:** Remover qualquer `resolutions` ou `overrides` manuais no `package.json` que forcem versões antigas.

### 3.3. Refatoração do Singleton (Fase 3)
Refatorar `apps/api/src/lib/prisma.ts` para suportar o Adaptador Neon:
```typescript
// Exemplo de implementação robusta esperada
import { Pool, neonConfig } from '@neondatabase/serverless';
import { PrismaNeon } from '@prisma/adapter-neon';
import ws from 'ws';

neonConfig.webSocketConstructor = ws;
const connectionString = env.DATABASE_URL;
const pool = new Pool({ connectionString });
const adapter = new PrismaNeon(pool);
const prisma = new PrismaClient({ adapter });
```

### 3.4. Validação de QA (Fase 4)
- **Teste de Latência:** Rodar `src/modules/platform/controllers/health.controller.ts` e medir o tempo de resposta do DB. Alvo: `< 100ms`.
- **Teste de Singleton (H-1):** Monitorar `pg_stat_activity` no Neon durante o `pnpm test` para garantir que o número de conexões não explode.

## 4. Critérios de Aceitação (Acceptance Criteria)

### AC1 — Sincronia de Versão (P0)
- [ ] Versão `7.x` (latest stable) presente em todos os `package.json` do monorepo.
- [ ] `pnpm list prisma` retorna a mesma versão em todos os workspaces.

### AC2 — Driver Adapter Neon (P0)
- [ ] O `PrismaClient` na API está utilizando o `adapter` injetado (Neon via WebSocket).
- [ ] O driver nativo `postgresql` (TCP) não é instanciado em ambiente de produção/cloud.

### AC3 — Estabilidade e Tipagem (P1)
- [ ] `pnpm typecheck` verde em todo o monorepo (especialmente nos serviços que usam `Prisma.decimal`).
- [ ] Migração de `$transaction` API concluída se houver breaking changes detectadas.

### AC4 — Performance Benchmarking (P1)
- [ ] Tempo de resposta de queries de leitura simples reduzido em pelo menos 50% em relação à baseline da v5 (TCP).
- [ ] Logs de boot da API confirmam inicialização bem-sucedida do adaptador Neon.
