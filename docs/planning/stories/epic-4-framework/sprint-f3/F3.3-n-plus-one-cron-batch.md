## Contexto

Auditoria de 2026-04-20 identificou padrão N+1 em dois serviços de cron jobs que rodam periodicamente contra o banco de produção (Neon PostgreSQL):

- **`lease.service.ts`** (linhas 186–216): itera sobre contratos de locação e, para cada item, executa `findFirst` seguido de `create` dentro do loop — gerando 2N queries para N registros. Em propriedades com muitos contratos (cenário real em property management), esse padrão bloqueia o pool de conexões e aumenta a latência do cron.
- **`recurring.service.ts`** (linhas 109–149): itera sobre transações recorrentes e executa `update` + `create` individualmente por registro — mesma classe de problema.

Ambos são jobs assíncronos que não precisam de resposta em tempo real, mas rodando contra Neon com limite de conexões em serverless, queries em loop causam latência acumulada e risco de timeout. A correção substitui os loops por operações em batch com Prisma (`createMany`, `upsertMany` via raw SQL, ou `$transaction` com array de operations).

---

## Escopo

### 1. `lease.service.ts` — loop findFirst+create → batch upsert

**Arquivo:** `apps/api/src/modules/property-management/services/lease.service.ts:186-216`

- Ler o loop existente e mapear:
  - Qual collection está sendo iterada (ex: lista de contratos ativos).
  - Qual condição do `findFirst` (chave de unicidade para o upsert).
  - Quais campos são escritos no `create`.
- Refatorar para:
  1. Coletar todos os registros necessários antes do loop (uma query).
  2. Computar os dados a criar/atualizar em memória (sem I/O por item).
  3. Executar um único `prisma.X.createMany({ data: [...], skipDuplicates: true })` ou `prisma.$transaction([...])` com array de operations.
- Se a lógica exigir upsert real (insert or update), usar `prisma.$queryRaw` com `INSERT ... ON CONFLICT DO UPDATE` ou `$transaction` com array de `upsertAsync` calls (ainda N queries mas dentro de uma transação — não N round-trips independentes).
- Documentar a abordagem escolhida em comentário inline se a decisão não for óbvia.

### 2. `recurring.service.ts` — loop update+create → batch

**Arquivo:** `apps/api/src/modules/finance/services/recurring.service.ts:109-149`

- Mesma análise: mapear collection iterada, condições, campos escritos.
- Separar os registros em dois grupos em memória: os que precisam de `update` e os que precisam de `create`.
- Executar `prisma.X.updateMany` para o grupo de updates (se todos compartilham o mesmo set de campos a atualizar — caso contrário, `$transaction` com array).
- Executar `prisma.X.createMany` para o grupo de creates.
- Garantir que a lógica de negócio (ex: datas de vencimento, geração de ID) seja computada antes do batch, não dentro de callbacks individuais.

### 3. Testes

- Para cada serviço: adicionar (ou atualizar) teste unitário que:
  - Passa N itens no mock (ex: N=10).
  - Verifica que o Prisma mock recebeu 1 chamada de batch (não N chamadas individuais).
  - Verifica que os dados persistidos estão corretos para todos os N itens.
- Se já existem testes cobrindo esses caminhos: confirmar que passam após a refatoração.

## Out of scope

- Refatoração arquitetural dos cron jobs para usar workers/queues (BullMQ, etc.) — decisão separada.
- Outros loops com queries em serviços além dos dois mapeados — criar issues separados se encontrados.
- Otimização de índices do Prisma/PostgreSQL para as queries em questão — escopo de uma story de database tuning.
- Migração para Prisma `$extends` — não necessário para este fix.
