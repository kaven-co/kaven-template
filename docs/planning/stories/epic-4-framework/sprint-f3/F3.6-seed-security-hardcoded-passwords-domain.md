## Contexto

Auditoria técnica de 2026-04-20 identificou dois findings no arquivo de seed (`packages/database/prisma/seed.ts`) que expõem credenciais e configurações de infraestrutura hardcoded:

- **H-4**: As personas de Finance, Support, Marketing e DevOps têm fallback `"ChangeMe123!"` quando a variável de ambiente correspondente não está configurada. O seed de ADMIN (linha 265) lança erro corretamente se a env não existir — as personas não seguem o mesmo padrão, criando uma assimetria silenciosa onde usuários técnicos são criados com senha conhecida publicamente sem qualquer aviso.
- **L-4**: `domain: "admin.kaven.dev"` está hardcoded na linha 209. Qualquer ambiente diferente de produção (staging, local, smoke test) cria o tenant com o domain errado, quebrando routing e introduzindo dados fictícios no banco.

Ambos os fixes são cirúrgicos e não afetam a API pública do seed.

---

## Escopo

### 1. H-4 — Personas sem fallback inseguro

**Arquivo:** `packages/database/prisma/seed.ts:417-441`

- Identificar cada persona (Finance, Support, Marketing, DevOps) que usa fallback `"ChangeMe123!"`.
- Substituir o fallback por `throw new Error('Env var <NOME_VAR> is required for seed persona <ROLE>')` — mesmo padrão do ADMIN na linha 265.
- Alternativa aceitável se há necessidade de seed local sem env: usar `crypto.randomBytes(16).toString('hex')` como senha temporária e **logar um warning** explícito: `console.warn('[SEED] Generated random password for <ROLE> — set <ENV_VAR> for reproducible seeds')`.
- Documentar no comentário do bloco qual variável de ambiente cada persona espera.

### 2. L-4 — Domain controlado por variável de ambiente

**Arquivo:** `packages/database/prisma/seed.ts:209`

- Localizar `domain: "admin.kaven.dev"`.
- Substituir por `domain: process.env.SEED_ADMIN_DOMAIN ?? "admin.kaven.dev"` com log de warning se a variável não estiver definida: `console.warn('[SEED] SEED_ADMIN_DOMAIN not set, using default admin.kaven.dev')`.
- Adicionar `SEED_ADMIN_DOMAIN` ao `.env.example` do workspace `packages/database/` (ou raiz) com comentário explicativo.
- Verificar se há outras ocorrências de `kaven.dev` hardcoded no seed; se houver, aplicar o mesmo padrão.

## Acceptance Criteria

### AC1 — Personas sem senha hardcoded conhecida
- [ ] Nenhuma persona em `seed.ts` contém a string `"ChangeMe123!"` como valor de senha.
- [ ] Personas sem env configurada: ou lançam erro explícito, ou geram senha aleatória com warning no stdout.
- [ ] `grep -n "ChangeMe123" packages/database/prisma/seed.ts` retorna zero resultados.

### AC2 — Domain via variável de ambiente
- [ ] `seed.ts:209` (ou linha equivalente) não contém `"admin.kaven.dev"` hardcoded.
- [ ] Variável `SEED_ADMIN_DOMAIN` documentada em `.env.example`.
- [ ] Seed executado sem a variável: funciona com domain default e emite warning no stdout.
- [ ] Seed executado com `SEED_ADMIN_DOMAIN=admin.custom.local`: tenant criado com domain correto.

### AC3 — Quality gates
- [ ] `pnpm lint` verde no workspace `packages/database`.
- [ ] `pnpm typecheck` verde no workspace `packages/database`.
- [ ] Seed roda sem erros em ambiente local com `.env` padrão.

## Out of scope

- Rotação de senhas existentes no banco de produção — operação de infra/owner.
- Auditoria de hardcoded strings em outros arquivos fora de `seed.ts`.
- Implementação de seed parametrizado completo (múltiplos ambientes) — escopo desta story é apenas os dois findings pontuais.
- Alteração do schema do banco ou das migrations.
