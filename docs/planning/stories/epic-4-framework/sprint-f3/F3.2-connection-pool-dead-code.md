## Contexto

Dois findings independentes mas relacionados ao ciclo de vida do Prisma na API:

- **H-1**: `authorization.service.ts` instancia `new PrismaClient()` diretamente em vez de usar o singleton exportado por `lib/prisma`. Cada instância cria um connection pool separado. Em produção (Cloud Run), onde instâncias escalam horizontalmente com cold-starts frequentes, isso multiplica conexões abertas ao Neon PostgreSQL sem que sejam liberadas corretamente — aproximando o limite de conexões do pool.
- **H-8**: `prisma-soft-delete.ts` existe no diretório de middleware mas não tem nenhum importador em toda a codebase. O arquivo nunca foi integrado ao pipeline de request do Fastify. Manter código morto induz risco de integração futura incorreta (alguém importa assumindo que está ativo) e gera confusão na leitura da arquitetura.

---

## Escopo

### 1. H-1 — Substituir `new PrismaClient()` pelo singleton

**Arquivo:** `apps/api/src/services/authorization.service.ts:34`

- Remover a instanciação local `new PrismaClient()`.
- Importar o singleton: `import { prisma } from '../lib/prisma'` (ajustar path conforme posição relativa do arquivo).
- Substituir todas as referências à variável local pelo `prisma` importado.
- Verificar se o construtor ou função de inicialização do service faz `prisma.$disconnect()` — remover se existir (o singleton não deve ser desconectado por consumers individuais).
- Rodar `grep -rn "new PrismaClient" apps/api/src/ --include="*.ts"` após a mudança — zero resultados fora de `lib/prisma.ts` é o gate.

### 2. H-8 — Deletar `prisma-soft-delete.ts`

**Arquivo:** `apps/api/src/middleware/prisma-soft-delete.ts`

- Confirmar zero importadores antes de deletar:
  ```bash
  grep -rn "prisma-soft-delete" apps/api/src/ --include="*.ts"
  ```
  Esperado: zero resultados.
- Deletar o arquivo.
- Se o soft-delete global for necessário no futuro, a abordagem correta é Prisma middleware/extension registrado em `lib/prisma.ts` — não middleware Fastify. Documentar essa decisão no PR description.
