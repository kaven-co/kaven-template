## Contexto

Auditoria técnica de 2026-04-20 identificou que a feature de documentos está completamente quebrada em produção:

- **L-2**: `apps/api/src/modules/documents/services/storage.service.ts:81` — método de upload retorna URL falsa com header `X-Placeholder=true`.
- **L-2**: `apps/api/src/modules/documents/services/storage.service.ts:109` — método de download retorna URL falsa com o mesmo header.

Qualquer tenant que tenta fazer upload ou download de documento recebe uma URL que não funciona. A feature está silenciosamente quebrada — não retorna erro 500, apenas uma URL placeholder, o que dificulta diagnóstico pelo usuário.

O framework já tem infraestrutura S3 configurada (usada pelo marketplace para artefatos de módulos). A integração com S3/GCS para presigned URLs é o caminho natural.

---

## Escopo

### 1. Verificar configuração S3 existente

- Localizar as env vars de S3 em `apps/api/src/config/env.ts`:
  ```
  grep -n "S3\|GCS\|BUCKET\|STORAGE" apps/api/src/config/env.ts
  ```
- Se não existirem variáveis de storage para o framework (apenas para o marketplace), adicionar ao schema Zod:
  ```typescript
  STORAGE_BUCKET: z.string().min(1).optional(),
  STORAGE_REGION: z.string().default('us-east-1'),
  STORAGE_ENDPOINT: z.string().url().optional(), // Para GCS via interoperabilidade S3
  AWS_ACCESS_KEY_ID: z.string().min(1).optional(),
  AWS_SECRET_ACCESS_KEY: z.string().min(1).optional(),
  ```
- Documentar as variáveis necessárias em `.env.example`.

### 2. Implementar upload presigned URL (PUT)

**Arquivo:** `apps/api/src/modules/documents/services/storage.service.ts:81`

- Substituir o placeholder por geração real de presigned PUT URL usando `@aws-sdk/s3-request-presigner`:
  ```typescript
  import { S3Client, PutObjectCommand } from '@aws-sdk/client-s3';
  import { getSignedUrl } from '@aws-sdk/s3-request-presigner';

  // Key pattern: documents/{tenantId}/{documentId}/{filename}
  const command = new PutObjectCommand({
    Bucket: env.STORAGE_BUCKET,
    Key: `documents/${tenantId}/${documentId}/${filename}`,
    ContentType: contentType,
  });
  const url = await getSignedUrl(s3Client, command, { expiresIn: 3600 });
  ```
- Remover o header `X-Placeholder=true`.
- Garantir que a key inclui `tenantId` para isolamento multi-tenant.

### 3. Implementar download presigned URL (GET)

**Arquivo:** `apps/api/src/modules/documents/services/storage.service.ts:109`

- Substituir o placeholder por geração real de presigned GET URL:
  ```typescript
  import { GetObjectCommand } from '@aws-sdk/client-s3';

  const command = new GetObjectCommand({
    Bucket: env.STORAGE_BUCKET,
    Key: storageKey, // Extraído do registro no banco
  });
  const url = await getSignedUrl(s3Client, command, { expiresIn: 900 }); // 15 min
  ```
- Remover o header `X-Placeholder=true`.

### 4. Verificar se `@aws-sdk/client-s3` e `@aws-sdk/s3-request-presigner` estão em `dependencies`

- `grep "@aws-sdk" apps/api/package.json`
- Se não estiver: `pnpm add @aws-sdk/client-s3 @aws-sdk/s3-request-presigner --filter api`
- Se o marketplace já usa: verificar se há um shared client que pode ser reutilizado.

### 5. Testes

- Adicionar testes unitários com mock do S3 client:
  - `generateUploadUrl()` → retorna URL que começa com `https://` e inclui o key correto.
  - `generateDownloadUrl()` → idem.
  - Storage não configurado (env vars ausentes) → retorna erro explícito, não URL falsa.
- Smoke test manual documentado no PR: upload de arquivo via presigned URL → confirmar objeto no bucket → download via presigned URL → confirmar conteúdo.
