# F3.11 — Module Structure: flat modules + requireCapability enforcement (M-9, M-10)

**Status:** READY
**Sprint:** F3
**Epic:** EPIC-004 Framework
**Depends on:** —
**Blocks:** —
**Created:** 2026-04-20
**Effort:** 4–6h

---

## Contexto

Dois findings de consistência arquitetural identificados na auditoria de 2026-04-20:

- **M-9**: Os módulos `checkout`, `billing-accounts` e `usage` têm estrutura flat — a lógica de negócio está diretamente em `routes.ts`, sem separação em `controllers/` e `services/`. Os outros 43+ módulos do framework seguem a estrutura padrão `controllers/ + services/ + routes/`. A inconsistência dificulta onboarding, testes unitários (services não são testáveis isoladamente) e manutenção.
- **M-10**: O middleware `requireCapability` está presente no framework e é a forma canônica de enforcement de feature limits — mas está aplicado em apenas 5 de 43+ módulos. Módulos premium sem enforcement ficam acessíveis a qualquer tenant independente de plano, o que é um bug de negócio além de uma inconsistência técnica.

---

## Escopo

### 1. M-9 — Refatorar módulos flat para estrutura padrão

**Módulos afetados:** `checkout`, `billing-accounts`, `usage`

Para cada módulo:

1. Criar diretório `controllers/` dentro do módulo.
2. Criar diretório `services/` dentro do módulo (se não existir).
3. Extrair lógica de negócio de `routes.ts` para `services/<module>.service.ts`.
4. Extrair handlers de request/response de `routes.ts` para `controllers/<module>.controller.ts`.
5. Refatorar `routes.ts` para apenas registrar rotas e injetar controller (sem lógica).
6. Atualizar imports em `app.ts` se necessário (não deve mudar — apenas o caminho interno muda).
7. Garantir que os testes existentes continuam passando após refatoração.

**Estrutura esperada após refatoração:**
```
modules/
  checkout/
    controllers/
      checkout.controller.ts
    services/
      checkout.service.ts
    routes.ts          ← apenas registro de rotas + injeção de controller
  billing-accounts/
    controllers/
      billing-accounts.controller.ts
    services/
      billing-accounts.service.ts
    routes.ts
  usage/
    controllers/
      usage.controller.ts
    services/
      usage.service.ts
    routes.ts
```

### 2. M-10 — Mapear e aplicar requireCapability

**Etapa 1 — Auditoria:**
- Executar grep para identificar todos os módulos e quais têm `requireCapability`:
  ```
  grep -rn "requireCapability" apps/api/src/modules/
  ```
- Gerar mapa: módulo → capability → plano mínimo requerido.

**Etapa 2 — Classificação:**
- Categorizar módulos por tier:
  - **Starter (todos os planos):** não precisa de gate.
  - **Complete/Pro:** adicionar `requireCapability('FEATURE_X')` nas rotas protegidas.
  - **Enterprise only:** idem com capability específica.
- Consultar `packages/database/prisma/seeds/capabilities.seed.ts` (ou equivalente) para o mapeamento de capabilities × planos existente.

**Etapa 3 — Implementação:**
- Para cada módulo identificado como premium sem enforcement:
  - Adicionar `requireCapability` nas rotas relevantes (não necessariamente todas as rotas — ex: GET pode ser livre, POST/DELETE pode requerer capability).
  - Adicionar testes: request com tenant sem capability → 403. Request com tenant com capability → 200.

**Nota:** Esta etapa pode ser grande. Se houver mais de 15 módulos sem enforcement, criar sub-tasks ou limitar o escopo desta story aos módulos de maior risco de negócio (billing, payments, AI features).

## Acceptance Criteria

### AC1 — Estrutura flat eliminada
- [ ] `checkout/routes.ts`, `billing-accounts/routes.ts`, `usage/routes.ts` contêm apenas registro de rotas — sem lógica de negócio diretamente no handler.
- [ ] Cada módulo tem `controllers/<module>.controller.ts` e `services/<module>.service.ts`.
- [ ] `pnpm test --filter api` passa sem regressões nos testes dos módulos refatorados.

### AC2 — Auditoria de requireCapability documentada
- [ ] Lista completa de módulos × capability × plano presente no PR description.
- [ ] Módulos sem enforcement classificados por risco.

### AC3 — requireCapability aplicado nos módulos de alto risco
- [ ] Pelo menos os módulos `checkout`, `billing-accounts` e `usage` (recém-refatorados) têm `requireCapability` nas rotas que criam/modificam recursos.
- [ ] Teste por módulo: tenant sem capability → 403; tenant com capability → 200.

### AC4 — Quality gates
- [ ] `pnpm test --filter api` — suite completa passando.
- [ ] `pnpm typecheck` verde.
- [ ] `pnpm lint` verde.

---

## Out of scope

- Criação de novas capabilities ou planos de pricing — apenas usar capabilities existentes.
- Refatoração de outros módulos além dos 3 identificados (checkout, billing-accounts, usage).
- Mudança de schema ou migrations.
- Frontend enforcement (apenas backend).
