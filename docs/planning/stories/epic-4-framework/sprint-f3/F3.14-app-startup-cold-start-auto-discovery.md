# F3.14 — App Startup: reduzir cold start + auto-discovery de módulos (M-6, L-3)

**Status:** READY
**Sprint:** F3
**Epic:** EPIC-004 Framework
**Depends on:** F3.11
**Blocks:** —
**Created:** 2026-04-20
**Effort:** 4–6h

---

## Contexto

Dois findings relacionados ao startup da API identificados na auditoria de 2026-04-20:

- **M-6**: `apps/api/src/app.ts` faz 182 `register()` calls manuais durante o startup. No Cloud Run com cold start, cada `register()` síncrono adiciona latência ao boot. O arquivo tem 712 linhas, é o maior gargalo de manutenção do projeto e precisa ser tocado a cada novo módulo adicionado.
- **L-3**: Não há auto-discovery de módulos. A consequência direta do M-6: adicionar um novo módulo requer edição manual de `app.ts`, aumentando o risco de conflitos em merge, ordem de registro errada e esquecimento de registrar o plugin.

O Fastify tem suporte nativo a auto-loading via `@fastify/autoload`, que resolve ambos os problemas: carrega plugins automaticamente de um diretório, em paralelo quando possível, e elimina o registro manual.

---

## Escopo

### 1. Avaliar abordagem antes de implementar

Antes de qualquer mudança, documentar no PR a análise de:

**Opção A — `@fastify/autoload` completo:**
- Carrega todos os arquivos de um diretório como plugins Fastify automaticamente.
- Elimina os 182 `register()` calls.
- Requer que todos os módulos exportem um plugin Fastify válido (função async com `(fastify, opts)` signature).
- Risco: módulos com side effects no load ou dependências de ordem de registro podem quebrar.

**Opção B — Barrel file + registro dinâmico:**
- Criar `apps/api/src/modules/index.ts` que exporta array de todos os plugins.
- `app.ts` itera sobre o array e faz `register()` em loop.
- Menor risco de quebra de ordem, mantém controle explícito.
- Reduz `app.ts` de 712 linhas para ~50 linhas, mas ainda requer edição do barrel a cada novo módulo.

**Opção C — Híbrido: autoload para módulos de negócio, manual para infraestrutura:**
- `@fastify/autoload` para `src/modules/**` (negócio).
- Manual para `src/plugins/**` e `src/lib/**` (infraestrutura: DB, auth, cors, etc.) onde a ordem importa.
- Melhor balanço entre automação e controle.

**Recomendação:** Implementar Opção C. Documentar a decisão no PR.

### 2. Instalar dependência

```bash
pnpm add @fastify/autoload --filter api
```

### 3. Implementar autoload para módulos de negócio

**Arquivo:** `apps/api/src/app.ts`

- Identificar os blocos de `register()` que correspondem a módulos de negócio (em `src/modules/`).
- Substituir esses blocos por:
  ```typescript
  await fastify.register(autoload, {
    dir: path.join(__dirname, 'modules'),
    options: { prefix: '/api' },
    dirNameRoutePrefix: false,
  });
  ```
- Garantir que cada módulo exporta default como plugin Fastify (`export default fp(async (fastify) => { ... })`).
- Manter registro manual para: middlewares de auth, plugins de DB (Prisma), CORS, rate limiting, helmet, swagger — onde a ordem de registro é crítica.

### 4. Verificar comportamento de cold start

- Medir tempo de startup antes e depois:
  ```bash
  time node dist/server.js &
  curl http://localhost:3099/api/health/health
  ```
- Documentar baseline (antes) e resultado (depois) no PR.

### 5. Testes de regressão

- Garantir que todas as rotas existentes continuam acessíveis após a migração.
- Executar suite completa: `pnpm test --filter api`.
- Verificar que a ordem de registro de plugins de infraestrutura não foi alterada.

## Acceptance Criteria

### AC1 — app.ts reduzido
- [ ] `apps/api/src/app.ts` tem menos de 200 linhas (redução significativa das 712 atuais).
- [ ] Blocos de `register()` manuais para módulos de negócio substituídos por `@fastify/autoload`.

### AC2 — Auto-discovery funcional
- [ ] Adicionar um novo módulo em `src/modules/` sem editar `app.ts` faz o módulo ser registrado automaticamente.
- [ ] Documentado no PR: como adicionar novo módulo (exemplo com 3 steps).

### AC3 — Plugins de infraestrutura preservados
- [ ] Auth, DB, CORS, rate limiting, helmet mantêm registro manual e ordem explícita.
- [ ] Nenhum plugin de infraestrutura foi movido para autoload.

### AC4 — Sem regressões
- [ ] `pnpm test --filter api` — suite completa passando (~2100 tests).
- [ ] Todas as rotas testadas nos testes de integração continuam respondendo.
- [ ] `pnpm typecheck` verde.

### AC5 — Baseline de cold start documentado
- [ ] Tempo de startup antes e depois registrado no PR description.

## Out of scope

- Migração para arquitetura de microservices ou serverless functions.
- Lazy loading de módulos por rota (otimização avançada).
- Mudança no protocolo de comunicação entre módulos.
- Autoload para workspaces além de `apps/api`.
- Métricas de performance em produção (requer acesso ao Cloud Run).
