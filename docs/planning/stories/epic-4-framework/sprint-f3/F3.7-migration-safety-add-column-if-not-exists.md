# F3.7 — Migration Safety: ADD COLUMN sem IF NOT EXISTS (H-9)

**Blocks:** —

## Contexto

Auditoria técnica de 2026-04-20 identificou que a maioria das migrations históricas geradas pelo Prisma usam `ALTER TABLE ... ADD COLUMN` sem a cláusula `IF NOT EXISTS`. Exemplos afetados:

- `20260418030000_f2_4_1_billing_account_schema`
- `20260418040000_f2_4_3_1_billing_account_source`
- `20260325000000_add_projects_pm_module`

A migration `20260411000000_add_tenant_stripe_customer_id` é o modelo correto — foi criada manualmente com `IF NOT EXISTS` após um incidente anterior.

**Risco concreto:** Ao re-executar migrations em ambientes degradados (restore de backup parcial, re-deploy de branch, migração entre Neon branches), um `ADD COLUMN` sem `IF NOT EXISTS` falha com `column already exists` e bloqueia o `prisma migrate deploy` inteiro — sem rollback automático.

**Complicador técnico crítico:** O Prisma mantém um `checksum` SHA-256 de cada arquivo de migration na tabela `_prisma_migrations`. Editar o conteúdo de uma migration histórica invalida o checksum e faz o `prisma migrate deploy` falhar com `drift detected`. Portanto, a abordagem de edição direta dos arquivos `.sql` **não é viável** sem intervenção manual no banco.

---

## Escopo

### 1. Auditoria — Mapear migrations afetadas

- Executar grep em `packages/database/prisma/migrations/` para identificar todos os `ALTER TABLE ... ADD COLUMN` sem `IF NOT EXISTS`:
  ```
  grep -rn "ADD COLUMN" packages/database/prisma/migrations/ | grep -v "IF NOT EXISTS"
  ```
- Gerar lista completa com: nome da migration, tabela afetada, coluna adicionada.
- Documentar a lista no corpo da PR como evidence.

### 2. Decisão de abordagem — Documentar no PR

Avaliar e documentar a abordagem escolhida entre as opções:

**Opção A — Wrapper migration (recomendada):**
- Criar uma nova migration `YYYYMMDDXXXXXX_safety_add_if_not_exists_guard` com `DO $$ BEGIN ... EXCEPTION WHEN duplicate_column THEN null; END $$;` wraps para as colunas críticas.
- Não toca nos arquivos históricos — checksum preservado.
- Cobre apenas migrations de alto risco (identificadas na auditoria).

**Opção B — Documentação de procedimento operacional:**
- Criar `docs/runbooks/migration-recovery.md` com steps para re-aplicar migrations em caso de drift.
- Indicar explicitamente quais migrations são frágeis e como contornar manualmente.

**Opção C — Edição + reset manual do checksum (apenas para ambientes não-prod):**
- Aceito apenas em branches Neon de desenvolvimento, nunca em prod.
- Documentar o procedimento de atualização do checksum na tabela `_prisma_migrations`.

**Recomendação:** Implementar Opção A para as 3 migrations identificadas como mais críticas + Opção B como documentação permanente.

### 3. Padrão para migrations futuras

- Adicionar seção em `docs/architecture/coding-standards.md` ou equivalente documentando:
  - Migrations manuais DEVEM usar `IF NOT EXISTS` / `IF EXISTS` em todos os DDL statements.
  - Migrations geradas pelo Prisma devem ser revisadas antes do merge para adicionar guards quando necessário.
  - Template de migration segura com exemplos de `ADD COLUMN IF NOT EXISTS`, `DROP COLUMN IF EXISTS`, `CREATE INDEX IF NOT EXISTS`.

## Acceptance Criteria

### AC1 — Auditoria documentada
- [ ] Lista completa de migrations com `ADD COLUMN` sem `IF NOT EXISTS` presente na PR como comentário ou arquivo em `docs/`.
- [ ] Decisão de abordagem documentada com justificativa.

### AC2 — Mitigação implementada (conforme abordagem escolhida)
- [ ] Se Opção A: migration wrapper criada e `prisma migrate deploy` executa sem erros em banco limpo E em banco que já tem as colunas.
- [ ] Se Opção B: runbook criado em `docs/runbooks/migration-recovery.md` com steps verificados.
- [ ] Em qualquer opção: as 3 migrations identificadas como exemplos no finding estão cobertas.

### AC3 — Padrão documentado
- [ ] `docs/architecture/coding-standards.md` (ou equivalente) contém seção sobre migration safety com template de DDL seguro.

### AC4 — Quality gates
- [ ] `pnpm prisma validate` verde no workspace `packages/database`.
- [ ] `pnpm typecheck` verde.
- [ ] Nenhuma migration existente teve seu arquivo `.sql` editado (checksum preservado).

## Out of scope

- Edição direta de arquivos `.sql` históricos em produção — risco de drift e bloqueio de deploy.
- Migração de todos os 37 arquivos de migration existentes — escopo é cobertura das mais críticas + documentação de padrão.
- Implementação de migration runner customizado para substituir o Prisma.
- Análise de migrations de outros repos do ecossistema (marketplace, template).
