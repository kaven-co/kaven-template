## Contexto

Auditoria técnica de 2026-04-20 identificou que a migration `20260325000000_add_projects_pm_module` adiciona as colunas `contact_id` e `manager_id` na tabela `projects` sem criar os indexes correspondentes. Colunas usadas como FK ou como filtro em queries sem index causam sequential scans — o custo cresce linearmente com o volume de dados.

Em produção com múltiplos tenants, `projects` é uma tabela de alta cardinalidade. Queries como "listar projetos por manager" ou "projetos de um contact" sem index podem facilmente se tornar queries de 500ms+ com volume modesto (10k registros).

O finding também levanta a questão de outras tabelas de alta cardinalidade (`tasks`, `orders`, `invoices`) com possíveis FKs sem index.

---

## Escopo

### 1. Criar migration com indexes faltantes para `projects`

**Arquivo de migration a criar:** `packages/database/prisma/migrations/YYYYMMDDXXXXXX_add_indexes_projects_contact_manager/migration.sql`

Conteúdo da migration:
```sql
-- F3.12: Add missing FK indexes on projects table
CREATE INDEX IF NOT EXISTS "projects_contact_id_idx" ON "projects"("contact_id");
CREATE INDEX IF NOT EXISTS "projects_manager_id_idx" ON "projects"("manager_id");
```

- Usar `IF NOT EXISTS` (padrão estabelecido em F3.7).
- Executar `prisma migrate dev --name add_indexes_projects_contact_manager` para gerar a migration no Prisma.
- Verificar que o schema Prisma também reflete os indexes (`@@index` no model `Project`).

### 2. Auditoria de FKs sem index em tabelas de alta cardinalidade

**Tabelas a auditar:** `tasks`, `orders`, `invoices`, `projects`, `tenant_users`

Para cada tabela:
- Listar todas as colunas `_id` (FKs convencionais) no schema Prisma.
- Verificar quais têm `@@index` ou `@unique` no schema.
- Para as que não têm: avaliar se são usadas em `WHERE`, `JOIN` ou `ORDER BY` em queries críticas.

**Comando de auditoria no schema:**
```bash
grep -A5 "model Task\|model Order\|model Invoice\|model Project\|model TenantUser" \
  packages/database/prisma/schema.prisma | grep -E "_id|@@index"
```

### 3. Criar migrations adicionais se necessário

- Para cada FK sem index identificado como crítico na auditoria:
  - Criar migration individual com `CREATE INDEX IF NOT EXISTS`.
  - Atualizar o model Prisma com `@@index([columnName])`.
- Se o número de indexes faltantes for grande (>10), criar um único migration consolidado com comentários por tabela.

### 4. Documentar baseline

- Adicionar comentário no PR com: tabelas auditadas, FKs encontradas, indexes criados, FKs explicitamente avaliadas e descartadas (com justificativa).
