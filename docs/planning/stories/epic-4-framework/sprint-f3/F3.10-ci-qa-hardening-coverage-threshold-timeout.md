## Contexto

Dois findings de CI/CD identificados na auditoria de 2026-04-20:

- **M-7**: `apps/api/vitest.config.ts` não configura `thresholds` no bloco `coverage`. O CI atual reporta cobertura, mas não falha o build quando ela cai abaixo de um mínimo aceitável. Com ~2100 testes e múltiplos módulos críticos (auth, billing, RBAC), a ausência de gate de cobertura significa que um PR pode introduzir código não testado sem alertas.
- **M-8**: `.github/workflows/ci.yml` não tem `timeout-minutes` no job `quality-gates`. O padrão do GitHub Actions é 6 horas. Um test hang (deadlock de DB, loop infinito em mock, conexão TCP pendurada) pode bloquear runners e consumir minutos de CI da conta inteira por horas.

Ambas as mudanças são de configuração pura, sem impacto em código de produção.

---

## Escopo

### 1. M-7 — Coverage threshold em vitest.config.ts

**Arquivo:** `apps/api/vitest.config.ts`

- Localizar o bloco `coverage` na configuração.
- Adicionar `thresholds` com valores iniciais conservadores que reflitam o estado atual da codebase (não devem quebrar a suite atual):

```typescript
coverage: {
  // ... configuração existente ...
  thresholds: {
    statements: 70,
    branches: 65,
    functions: 70,
    lines: 70,
  },
}
```

- **Antes de commitar:** executar `pnpm test --coverage --filter api` e verificar o percentual atual de cobertura. Ajustar os thresholds para ficarem 2-3% abaixo do valor atual (para não quebrar a suite imediatamente e dar espaço para variação).
- Documentar no PR o percentual de cobertura atual como baseline.
- Criar comentário `TODO: aumentar para 80% após F3.11 (module structure refactor)` nos thresholds.

### 2. M-8 — Timeout no job quality-gates

**Arquivo:** `.github/workflows/ci.yml`

- Localizar o job `quality-gates` (ou nome equivalente do job principal de CI).
- Adicionar `timeout-minutes: 30` no nível do job:

```yaml
jobs:
  quality-gates:
    timeout-minutes: 30
    runs-on: ubuntu-latest
    # ... restante da configuração
```

- Verificar se há outros jobs sem `timeout-minutes` no mesmo workflow (e.g., `build`, `deploy`, `typecheck`). Se houver, adicionar timeouts adequados:
  - Jobs de lint/typecheck: `timeout-minutes: 10`
  - Jobs de test: `timeout-minutes: 30`
  - Jobs de build/deploy: `timeout-minutes: 20`
- Não alterar a lógica dos steps — apenas adicionar o campo `timeout-minutes`.

## Acceptance Criteria

### AC1 — Coverage threshold configurado
- [ ] `apps/api/vitest.config.ts` contém bloco `thresholds` dentro de `coverage`.
- [ ] Thresholds definidos para `statements`, `branches`, `functions` e `lines`.
- [ ] `pnpm test --coverage --filter api` passa com os thresholds definidos (sem falha por cobertura insuficiente).
- [ ] Valor atual de cobertura documentado no PR description como baseline.

### AC2 — Timeout no CI
- [ ] Job `quality-gates` em `.github/workflows/ci.yml` tem `timeout-minutes` definido (≤ 30).
- [ ] Nenhum job de CI sem `timeout-minutes` (pelo menos os jobs principais cobertos).

### AC3 — CI verde
- [ ] Push da branch com as mudanças → GitHub Actions executa e passa (green check).
- [ ] Nenhum step de CI quebrado pelas mudanças de configuração.

## Out of scope

- Aumento de cobertura de testes — esta story apenas configura o gate, não cria novos testes.
- Configuração de coverage para workspaces além de `apps/api` (admin, tenant) — escopo desta story é apenas a API.
- Mudanças no fluxo de CI além de timeout e coverage threshold.
- Integração com serviços externos de coverage (Codecov, Coveralls).
