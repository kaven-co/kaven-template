## Contexto

Três findings de higiene de dependências identificados na auditoria de 2026-04-20:

- **M-4**: TypeScript `^6.0.2` no root `package.json` vs `^5.3.3` em `apps/api/package.json` e `packages/database/package.json`. Versões diferentes de TypeScript no mesmo monorepo causam divergências sutis de inferência de tipos, erros de typecheck difíceis de reproduzir por workspace e incompatibilidades com ferramentas que resolvem `tsc` do root.
- **M-5**: `apps/api/package.json` lista `nodemailer`, `postmark` e `resend` em `dependencies`. Apenas `resend` está ativo em produção (decisão documentada no CLAUDE.md). Manter providers inativos em `dependencies` (não em `devDependencies`) aumenta o bundle de produção e cria surface de ataque desnecessária.
- **L-1**: `apps/api/src/lib/email/providers/ses.provider.ts` (868 linhas) está registrado no sistema de email mas nunca será o provider ativo (Resend é o atual). O arquivo cria confusão para novos devs e pode ser inadvertidamente ativado por configuração incorreta.

---

## Escopo

### 1. M-4 — Unificar TypeScript

**Arquivos:** `package.json` (root), `apps/api/package.json`, `packages/database/package.json`

- Determinar a versão canônica a adotar:
  - Se `^6.0.2` do root é intencional e estável: atualizar `apps/api` e `packages/database` para `^6.0.2`.
  - Se `^5.3.3` é mais seguro (menos breaking changes): rebaixar o root para `^5.3.3` e abrir issue separada para TS 6 upgrade com análise dedicada.
  - **Recomendação:** atualizar todos para `^5.7.x` (última minor estável da série 5) como passo intermediário seguro, com TS 6 como story separada.
- Após unificar: `pnpm install` + `pnpm typecheck` em todos os workspaces deve passar sem novos erros.
- Documentar a decisão de versão no PR description.

### 2. M-5 — Remover providers de email inativos de `dependencies`

**Arquivo:** `apps/api/package.json`

- Mover `nodemailer` e `postmark` de `dependencies` para `devDependencies` (se há testes unitários que os importam diretamente).
- Se não houver nenhum import ativo de `nodemailer` ou `postmark` em `apps/api/src/`:
  - Executar: `grep -rn "nodemailer\|postmark" apps/api/src/`
  - Se zero resultados: remover completamente dos `package.json`.
  - Se há imports em providers não-ativos: avaliar case-by-case.
- Garantir que `resend` permanece em `dependencies`.

### 3. L-1 — Avaliar e marcar ses.provider.ts

**Arquivo:** `apps/api/src/lib/email/providers/ses.provider.ts`

- Verificar se o arquivo é importado em algum lugar ativo:
  ```
  grep -rn "ses.provider\|SesProvider\|ses-provider" apps/api/src/
  ```
- Se não é importado em nenhum path de produção:
  - **Opção A (preferida):** Mover para `apps/api/src/lib/email/providers/_disabled/ses.provider.ts` com comentário no topo: `// DISABLED: SES provider — Resend is the active email provider. Kept for reference only.`
  - **Opção B:** Remover o arquivo inteiramente se não há valor histórico.
- Atualizar o arquivo de index/barrel de providers para não exportar/registrar o SES provider.
- Documentar no PR qual provider está ativo e por quê.
