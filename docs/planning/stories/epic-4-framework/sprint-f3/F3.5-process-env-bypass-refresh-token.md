## Contexto

Três findings relacionados à gestão de variáveis de ambiente e geração de tokens:

- **H-2 — `process.env.*` bypassando o schema Zod**: além do `STRIPE_SECRET_KEY` já coberto por F3.1 (C-2), existe uso disperso de `process.env.*` diretamente em `apps/api/src/` fora de `config/env.ts`. Isso significa que mudanças no nome de uma env var podem quebrar silenciosamente (TypeScript não detecta a ausência), e valores ausentes só causam erro em runtime, não no boot da aplicação.

- **M-1 — `ENCRYPTION_KEY` com fallback para `JWT_SECRET`**: `encryption.util.ts:17` usa `process.env.ENCRYPTION_KEY ?? process.env.JWT_SECRET` como chave de encriptação. Em ambientes onde `ENCRYPTION_KEY` não está configurada (ex: staging sem a variável), dados são encriptados com o mesmo secret usado para JWT — criando acoplamento de segurança. Uma exposição do JWT secret também expõe dados encriptados. A key de encriptação deve ser obrigatória e independente.

- **M-2 — `Math.random()` para geração de refreshToken**: `jwt.ts:32` usa `Math.random()` para gerar o refresh token. `Math.random()` não é criptograficamente seguro (PRNG baseado em seed do engine V8 — previsível em alguns contextos). Tokens de sessão devem usar `crypto.randomBytes()`.

## Escopo

### 1. H-2 — Grep e migração de `process.env` direto

**Auditoria:**
```bash
grep -rn "process\.env\." apps/api/src/ --include="*.ts" \
  | grep -v "config/env.ts" \
  | grep -v "spec\|test\|\.spec\."
```

Para cada ocorrência encontrada:
- Se é uma variável já declarada no schema Zod de `env.ts`: substituir `process.env.VAR` por `env.VAR` (importar `env` se necessário).
- Se é uma variável nova não declarada: adicioná-la ao schema Zod com o tipo correto (`z.string()`, `z.string().optional()`, etc.) e então usar `env.VAR`.
- Documentar no PR: lista de todas as ocorrências encontradas e como foram tratadas.

**Gate de conclusão:**
```bash
grep -rn "process\.env\." apps/api/src/ --include="*.ts" \
  | grep -v "config/env.ts" \
  | grep -v "spec\|test\|\.spec\."
```
Esperado: zero resultados.

### 2. M-1 — `ENCRYPTION_KEY` obrigatória no schema

**Arquivo:** `apps/api/src/utils/encryption.util.ts:17` e `apps/api/src/config/env.ts`

- Remover o fallback `?? process.env.JWT_SECRET` em `encryption.util.ts`.
- Adicionar `ENCRYPTION_KEY: z.string().min(32)` no schema Zod de `env.ts` (mínimo 32 chars para AES-256 derivado via PBKDF2, ou 64 chars se for hex raw).
- Usar `env.ENCRYPTION_KEY` no `encryption.util.ts` (importar `env`).
- Atualizar `.env.example` no repo para incluir `ENCRYPTION_KEY=` com instrução de geração (ex: `openssl rand -hex 32`).
- Verificar se testes que usam `encryption.util.ts` precisam configurar `ENCRYPTION_KEY` no environment de teste — ajustar mocks/setup conforme necessário.

**Impacto em produção:** Se `ENCRYPTION_KEY` não estiver setada no Cloud Run, a aplicação não vai subir (Zod validation error no boot). Isso é o comportamento correto — fail-fast em vez de usar fallback silencioso. Antes do deploy desta story, confirmar que a variável está configurada no Cloud Run environment.

### 3. M-2 — `crypto.randomBytes` para refreshToken

**Arquivo:** `apps/api/src/lib/jwt.ts:32`

- Localizar o `Math.random()` ou derivado usado para gerar o refresh token.
- Substituir por:
  ```typescript
  import { randomBytes } from 'crypto';
  // ...
  const refreshToken = randomBytes(32).toString('hex'); // 64 chars hex, 256 bits de entropia
  ```
- Confirmar que o comprimento do token gerado é compatível com o campo de storage (coluna no DB, campo no Redis, etc.) — se a coluna tem limite de comprimento, verificar que 64 chars cabem.
- Atualizar testes que mockam a geração do refresh token — se usam `Math.random()` no mock, ajustar para mock de `crypto.randomBytes`.

## Acceptance Criteria

### AC1 — Zero `process.env` direto fora de `env.ts`
- [ ] Grep `process\.env\.` em `apps/api/src/` (excluindo `config/env.ts`, specs e tests) retorna zero resultados.
- [ ] Todas as variáveis migradas estão declaradas no schema Zod de `env.ts`.
- [ ] Typecheck verde (variáveis tipadas corretamente via `env`).

### AC2 — `ENCRYPTION_KEY` obrigatória
- [ ] `env.ts` declara `ENCRYPTION_KEY: z.string().min(32)` (sem optional, sem fallback).
- [ ] `encryption.util.ts` usa `env.ENCRYPTION_KEY`, não `process.env.ENCRYPTION_KEY` nem `process.env.JWT_SECRET`.
- [ ] `.env.example` contém `ENCRYPTION_KEY=` com instrução de geração.
- [ ] Aplicação falha no boot se `ENCRYPTION_KEY` ausente (comportamento Zod padrão — não requer code change adicional, só verificar).
- [ ] Testes que usam `encryption.util.ts` configuram `ENCRYPTION_KEY` no ambiente de teste.

### AC3 — `crypto.randomBytes` para refreshToken
- [ ] `jwt.ts` não contém `Math.random()`.
- [ ] Refresh token gerado via `crypto.randomBytes(32).toString('hex')` (ou equivalente com entropia ≥ 256 bits).
- [ ] Testes de geração de token passando.

## Out of scope

- `process.env` em scripts de migration, seed ou utilitários fora de `apps/api/src/` — escopo desta story é restrito ao código da API.
- Rotação de dados já encriptados com o fallback `JWT_SECRET` — se houver dados em produção encriptados com JWT_SECRET como chave, isso requer um script de re-encriptação separado (avaliar no PR se há dados afetados).
- `process.env` nos frontends (`apps/admin`, `apps/tenant`) — escopo separado, Next.js tem sua própria convenção de env vars.
- Mudança do algoritmo de encriptação (AES-GCM, ChaCha20, etc.) — manter o algoritmo atual, só corrigir a gestão da key.

## Nota de deploy

Antes do merge em produção, garantir que `ENCRYPTION_KEY` está configurada no Cloud Run:
```bash
gcloud run services update kaven-api \
  --update-env-vars ENCRYPTION_KEY=$(openssl rand -hex 32) \
  --project kaven-prod \
  --region us-east1
```
Se já existir uma `ENCRYPTION_KEY` configurada, manter o valor existente (não rotacionar sem um plano de re-encriptação dos dados).
