## Contexto

Dois findings de qualidade com impacto diferente mas que se resolvem na mesma janela de trabalho:

- **H-6 — Duas bibliotecas de bcrypt coexistindo**: `apps/api/package.json` declara tanto `bcrypt` (binding nativo Node.js) quanto `bcryptjs` (implementação pura JS). Ter ambas é um risco: código que importa uma vs a outra pode ter comportamentos sutis diferentes (ex: timings de hash, compatibilidade de salts em ambientes sem binários nativos). Em Cloud Run (Linux x64), `bcrypt` nativo funciona normalmente, mas `bcryptjs` pode ter sido adicionado como fallback e nunca removido. Manter as duas cria confusão nos imports e inchaço no bundle.

- **H-5 — Tests com `describe.skip`**: foram identificados 5 blocos `describe.skip` em `test/compliance/` e `test/multi-tenant/`. Testes de compliance e multi-tenant são exatamente os que validam os invariantes de segurança e isolamento do sistema (IDOR, GDPR, cross-tenant data leaks). Tê-los skipados sem documentação formal significa que regressões nesses domínios podem passar no CI sem qualquer sinal.

## Escopo

### 1. H-6 — Escolher e unificar em uma biblioteca bcrypt

**Decisão a tomar antes de implementar:**

Avaliar qual biblioteca está em uso majoritário nos imports atuais:
```bash
grep -rn "from 'bcrypt'\|require('bcrypt')" apps/api/src/ --include="*.ts"
grep -rn "from 'bcryptjs'\|require('bcryptjs')" apps/api/src/ --include="*.ts"
```

**Regra de decisão:**
- Se `bcrypt` (nativo) está em uso majoritário → remover `bcryptjs` de `package.json` + atualizar os arquivos que importam `bcryptjs` para importar `bcrypt`.
- Se `bcryptjs` está em uso majoritário (improvável mas possível) → remover `bcrypt`, unificar em `bcryptjs`, documentar o motivo em `docs/adrs/` (ex: ambientes sem binários nativos).
- **Preferência padrão**: `bcrypt` nativo, pois o ambiente Cloud Run é Linux x64 e tem binários disponíveis. `bcryptjs` é ~3x mais lento.

**Implementação:**
- Remover o pacote perdedor do `apps/api/package.json` (e do `pnpm-lock.yaml` via `pnpm install`).
- Atualizar todos os imports para o pacote vencedor.
- Confirmar que os tipos (`@types/bcrypt` ou `@types/bcryptjs`) estão corretos — manter apenas os tipos do pacote escolhido.
- Rodar `pnpm install` + `pnpm typecheck` após a mudança.

### 2. H-5 — Investigar e tratar os 5 `describe.skip`

**Localizar os skips:**
```bash
grep -rn "describe\.skip\|xdescribe" apps/api/test/compliance/ apps/api/test/multi-tenant/ --include="*.ts"
```

Para cada bloco skipado, fazer um dos seguintes (documentar qual foi escolhido e por quê no PR):

**Opção A — Unskip e fazer passar:** Se o teste foi skipado por flakiness ou implementação incompleta que já foi resolvida, remover o `.skip` e garantir que passa.

**Opção B — Unskip com fix mínimo:** Se o teste falha por causa de um setup incorreto (ex: seed de dados, mock desatualizado), corrigir o setup sem alterar a assertion.

**Opção C — Documentar formalmente como "known gap":** Se o comportamento testado não está implementado ainda (ex: GDPR right-to-erasure não existe no produto), substituir o `describe.skip` por `describe.todo` com comentário explicando:
- Por que o comportamento não existe ainda.
- Qual story/epic o implementará.
- Estimativa de quando será unskipado.

**Opção D — Deletar o teste:** Se o teste valida comportamento que foi deliberadamente removido ou descartado (ex: feature nunca lançada), deletar com comentário no PR.

**Gate:** Ao final, zero `describe.skip` sem comentário de justificativa em `test/compliance/` e `test/multi-tenant/`. Toda decisão (A/B/C/D) deve ser explicitada no PR description por teste.

## Acceptance Criteria

### AC1 — Uma única biblioteca bcrypt
- [ ] `apps/api/package.json` contém apenas uma das duas: `bcrypt` ou `bcryptjs` (não ambas).
- [ ] Zero imports do pacote removido em `apps/api/src/`.
- [ ] `pnpm typecheck` verde.
- [ ] `pnpm test --filter api` passando (hashing/verification tests OK).

### AC2 — Skips documentados ou removidos
- [ ] Todos os 5 `describe.skip` foram tratados (A, B, C ou D).
- [ ] Zero `describe.skip` sem comentário de justificativa em `test/compliance/` e `test/multi-tenant/`.
- [ ] Qualquer teste unskipado (opção A ou B) passa no CI.
- [ ] Qualquer `describe.todo` inclui link para story ou epic responsável.

### AC3 — Quality gates
- [ ] `pnpm test --filter api` — suite completa passando.
- [ ] `pnpm lint` e `pnpm typecheck` verdes.
- [ ] PR description documenta, por teste skipado, qual opção foi aplicada e por quê.

## Out of scope

- Implementar os comportamentos ausentes que motivam os testes skipados (ex: GDPR erasure, cross-tenant isolation completo) — pertencem às suas próprias epics/stories.
- Migração de bcrypt para argon2 ou outro algoritmo — decisão arquitetural separada.
- Auditoria de `describe.skip` em outros diretórios de teste além de `compliance/` e `multi-tenant/`.
