## Contexto

Auditoria técnica realizada em 2026-04-20 identificou quatro findings que, isolados, parecem menores, mas representam superfícies de ataque reais em produção:

- **C-1**: `jwt.ts` imprime fingerprint do JWT secret em `console.log` — dado que o API roda em Cloud Run com logs visíveis no GCP, qualquer pessoa com acesso ao projeto pode derivar o secret via log stream.
- **C-2**: `external-api-monitor.service.ts` lê `process.env.STRIPE_SECRET_KEY` diretamente, bypassando o schema Zod de `env.ts`. Isso significa que a variável pode estar ausente em runtime sem ser detectada no boot — além de criar inconsistência de acesso a secrets.
- **H-3**: `decrypt()` em `encryption.util.ts` captura erros e retorna string vazia silenciosamente. Código que depende do retorno não detecta falha de decriptação — o dado corrompido ou adulterado passa como se fosse string válida.
- **H-10**: `kaven-ui-base/.gitignore` não cobre arquivos `.env*` — risco de commit acidental de credenciais em builds locais do design system standalone.

Todos os quatro são corrigíveis de forma cirúrgica, sem impacto em interfaces públicas.

## Escopo

### 1. C-1 — Remover `console.log` em `jwt.ts`

**Arquivo:** `apps/api/src/lib/jwt.ts:41`

- Localizar o `console.log` que exibe fingerprint ou qualquer derivado do `JWT_SECRET`.
- Remover a linha inteiramente (não substituir por `secureLog` — nenhum log sobre o secret é aceitável nesse contexto).
- Verificar se há outros `console.log` no mesmo arquivo; se existirem, substituir por `secureLog` ou remover conforme caso.

### 2. C-2 — Mover `STRIPE_SECRET_KEY` para `env.ts`

**Arquivo:** `apps/api/src/modules/observability/services/external-api-monitor.service.ts:240-246`

- Identificar o bloco `process.env.STRIPE_SECRET_KEY` (leitura direta).
- Importar `env` de `config/env.ts` no arquivo.
- Substituir `process.env.STRIPE_SECRET_KEY` por `env.STRIPE_SECRET_KEY`.
- Confirmar que `STRIPE_SECRET_KEY` já existe no schema Zod de `env.ts`; se não existir, adicioná-la como `z.string().min(1)`.
- Rodar `grep -rn "process\.env\.STRIPE" apps/api/src/ --include="*.ts"` — zero resultados fora de `config/env.ts` é o gate.

### 3. H-3 — `decrypt()` deve propagar erro

**Arquivo:** `apps/api/src/utils/encryption.util.ts:71-74`

- Localizar o bloco `try/catch` em `decrypt()` que retorna `''` no catch.
- Substituir o retorno silencioso por `throw` do erro original (ou wrap com mensagem contextual: `throw new Error('Decryption failed', { cause: err })`).
- Verificar todos os call-sites de `decrypt()` em `apps/api/src/` — nenhum deles deve depender de retorno vazio como sentinel de falha; ajustar se necessário para capturar o erro explicitamente.
- Adicionar teste unitário: `decrypt()` com payload inválido → deve lançar erro (não retornar `''`).

### 4. H-10 — `.gitignore` do kaven-ui-base

**Arquivo:** `kaven-ui-base/.gitignore` (ou `packages/ui/.gitignore` se monorepo)

- Adicionar as seguintes entradas:

```
# Env files — never commit credentials
*.env
.env
.env.*
.env.local
.env.*.local
```

- Verificar se existe `.env.example` no diretório; se existir, garantir que `.env.example` NÃO está coberto pelo gitignore (deve ser versionado).

## Acceptance Criteria

- [x] `apps/api/src/lib/jwt.ts` não contém nenhum `console.log`.
