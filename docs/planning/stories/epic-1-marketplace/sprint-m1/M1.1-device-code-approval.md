# M1.1 — Device Code Approval Endpoint

> **Epic:** 1 (Marketplace) | **Sprint:** M1 | **Agent:** @dev
> **Priority:** P0 — BLOCKER (sem isso, auth CLI não funciona)
> **Depends on:** Nenhum

---

## Contexto

O fluxo Device Code (RFC 8628) está **incompleto**. O CLI gera um device code via `POST /auth/device-code` e faz polling via `POST /auth/token`, mas **nada no sistema marca o device code como APPROVED**. O campo `status` no DB fica `PENDING` para sempre.

Precisamos de:
1. Uma rota que exiba os detalhes do device code (para o frontend de ativação)
2. Uma rota que aprove ou negue o device code (ação do usuário autenticado)

## User Story

**Como** desenvolvedor usando o kaven-cli,
**Quero** que o device code que recebi no terminal seja aprovado quando eu acessar a URL de verificação e confirmar no browser,
**Para que** o CLI receba o JWT e eu possa usar os comandos autenticados.

---

## Acceptance Criteria

- [ ] `GET /auth/device/:userCode` retorna detalhes do device code (status, expiresAt, scopes)
- [ ] Retorna 404 se userCode não existe
- [ ] Retorna 410 (Gone) se expirado
- [ ] `POST /auth/device/:userCode/approve` marca status como `APPROVED` e associa userId
- [ ] `POST /auth/device/:userCode/deny` marca status como `DENIED`
- [ ] Aprovação requer autenticação (JWT válido no header)
- [ ] Após aprovação, o `POST /auth/token` polling do CLI retorna tokens
- [ ] Device codes expirados não podem ser aprovados (retorna 410)
- [ ] Device codes já consumidos não podem ser re-aprovados (retorna 409 Conflict)
- [ ] Respostas de erro seguem formato RFC 6749 (ver M1.4)

## Technical Notes

### Arquivo existente a modificar:

### Novas rotas:

```typescript
// GET /auth/device/:userCode — Buscar detalhes (para frontend mostrar "Approve this device?")
GET /auth/device/:userCode
→ 200: { user_code, status, expires_at, client_info }
→ 404: { error: "not_found" }
→ 410: { error: "expired_token" }

// POST /auth/device/:userCode/approve — Usuário autenticado aprova
POST /auth/device/:userCode/approve
Headers: Authorization: Bearer <jwt>
→ 200: { status: "approved", message: "Device authorized" }
→ 401: { error: "unauthorized" }
→ 409: { error: "already_used" }
→ 410: { error: "expired_token" }

// POST /auth/device/:userCode/deny — Usuário autenticado nega
POST /auth/device/:userCode/deny
→ 200: { status: "denied" }
```

### Lógica no AuthService:

```typescript
// Adicionar ao auth.service.ts
async getDeviceCodeByUserCode(userCode: string) {
  const code = await prisma.deviceCode.findFirst({ where: { userCode } });
  if (!code) throw new NotFoundError();
  if (new Date() > code.expiresAt) throw new ExpiredError();
  return code;
}

async approveDeviceCode(userCode: string, userId: string) {
  const code = await this.getDeviceCodeByUserCode(userCode);
  if (code.status !== 'PENDING') throw new ConflictError();
  return prisma.deviceCode.update({
    where: { id: code.id },
    data: { status: 'APPROVED', userId }
  });
}

async denyDeviceCode(userCode: string) {
  const code = await this.getDeviceCodeByUserCode(userCode);
  if (code.status !== 'PENDING') throw new ConflictError();
  return prisma.deviceCode.update({
    where: { id: code.id },
    data: { status: 'DENIED' }
  });
}
```

### Schema Prisma (já existe, não precisa alterar):
- `DeviceCode.status` já é String com default `PENDING`
- `DeviceCode.userId` já é nullable
