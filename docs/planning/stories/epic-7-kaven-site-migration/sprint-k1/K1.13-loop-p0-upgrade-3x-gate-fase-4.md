# K1.13 — Loop de correção (P0) via framework + kaven upgrade (>=3x) + Gate obrigatório Fase 4

> **Epic:** 7 (Kaven Site Production Migration) | **Sprint:** K1 | **Agent:** @architect (Atlas) + @qa (Shield)  
> **Priority:** P0 — Sem isso, não existe dogfooding real nem confiança de evolução.  
> **Depends on:** K1.11, K1.12

---

## Contexto

O “mecanismo central” do plano é o loop:
problema no `kaven-site` -> corrigir no `kaven-framework` -> release -> `kaven upgrade` -> validar em produção.

Esta story operacionaliza isso até **zerar P0** e provar **>=3 upgrades** end-to-end.

---

## User Story

**Como** responsável por estabilizar o produto,  
**Quero** corrigir P0s via framework e validar upgrades repetidos,  
**Para que** o `kaven-site` evolua com segurança e clientes herdem fixes automaticamente.

---

## Acceptance Criteria

### Triagem
- [ ] Lista consolidada de problemas da K1.11 e K1.12.
- [ ] Classificação: P0 / P1 / P2.
- [ ] Ordem de execução P0 definida por impacto no fluxo crítico.

### Loop (para cada P0)
- [ ] Root cause identificado no `kaven-framework` (não no `kaven-site_`).
- [ ] Fix implementado no framework com teste local mínimo (unit/integration quando aplicável).
- [ ] Release publicada (versão registrada).
- [ ] `kaven upgrade` executado no `kaven-site_`.
- [ ] Deploy (API + front) realizado.
- [ ] Fix validado em produção com evidência objetiva.

### Meta-gate
- [ ] `kaven upgrade` validado >= 3 vezes (ciclos reais, versões diferentes).
- [ ] Zero P0 remanescente em produção.
- [ ] P1 documentados e não bloqueiam.
- [ ] P2 enviados ao backlog.

### Parada obrigatória (Fase 4)
- [ ] Mensagem formal emitida: “🛑 PARADA OBRIGATÓRIA — FASE 4: LANDING PAGE”.
- [ ] Confirmação de que **não inicia landing** sem autorização explícita do Chris.

---

## Evidence

- [ ] Tabela de iterações (problema, causa raiz, fix, versão, validação) preenchida.
- [ ] Evidência de cada upgrade (versão antes/depois, output do comando).
- [ ] Evidência de validação em produção (curl, logs, screenshots, eventos Stripe quando aplicável).

---

## Definition of Done

- [ ] Zero P0 em produção.
- [ ] >=3 upgrades validados end-to-end.
- [ ] Gate Fase 4 disparado e registrado.

---

## Time Estimate

- Triagem + backlog: 2h
- 3 ciclos de upgrade (mínimo): 6h
- Correções P0 (variável): 6h
- Evidências + documentação final: 2h
- **Total: 16h** (não inclui implementações grandes fora do escopo)
