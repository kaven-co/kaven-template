# K1.7 — Configurar envs de produção (Vercel admin+tenant + Cloud Run API) com 4-eyes

> **Epic:** 7 (Kaven Site Production Migration) | **Sprint:** K1 | **Agent:** @devops (Deploy)  
> **Priority:** P0 — Config errada derruba auth/billing. Exige revisão 4-eyes.  
> **Depends on:** K1.1, K1.2, K1.6

---

## Contexto

Após o bootstrap, é necessário preparar o ambiente de produção para o `kaven-site` (admin/tenant/API) reaproveitando Neon/Upstash/Stripe/Resend e ajustando OAuth callbacks conforme plano.

---

## User Story

**Como** operador de produção,  
**Quero** configurar envs corretamente com revisão,  
**Para que** o deploy seja previsível e a validação de negócio funcione sem “caça ao env”.

---

## Acceptance Criteria

### Vercel
- [ ] Envs do projeto `admin` configurados para o `kaven-site` (prod + preview se necessário).
- [ ] Envs do projeto `tenant` configurados para o `kaven-site`.
- [ ] Variáveis críticas confirmadas: `API_URL`, `NEXTAUTH_URL`, `NEXTAUTH_SECRET`, `STRIPE_*`, `GITHUB_*`, `RESEND_*`.

### Cloud Run (API)
- [ ] Envs do serviço de API preparados para a nova imagem.
- [ ] `DATABASE_URL` aponta para Neon existente (kaven_framework).
- [ ] `REDIS_URL` aponta para Upstash existente.
- [ ] `STRIPE_WEBHOOK_SECRET` e publishable/secret keys coerentes.

### 4-eyes + segurança
- [ ] Checklist 4-eyes executado (executor + reviewer).
- [ ] Nenhum secret escrito em `.env` versionado.
- [ ] Plano de rotação/cutover aplicado quando necessário (JWT/NEXTAUTH/Stripe/OAuth).

---

## Evidence

- [ ] “Diff” de nomes de envs aplicado (sem valores).
- [ ] Registro de revisão 4-eyes.

---

## Definition of Done

- [ ] Ambientes prontos para deploy (API/admin/tenant) sem pendências críticas.

---

## Time Estimate

- Preparar Vercel envs: 2h
- Preparar Cloud Run envs: 2h
- Revisão 4-eyes + evidências: 1h
- **Total: 5h**
